Legal & Compliance Architecture

Status: Foundational — schema landed BL-129 (D-289). Ingestion and posture features phased through BL-130 → BL-140. Owner: Will (Product), Claude (Architecture), CC (Implementation) Last updated: 2026-04-30 Atlas section: legal-compliance/

TL;DR

HouseZoom provides decision support, not legal authority. Every compliance message we surface to an agent must:

  1. Cite an authoritative source (notaire guide, RERA manual, Légifrance article, etc.)
  2. Carry a verify-yourself disclaimer (locale-aware)
  3. Be reviewable, acceptable, or rejectable by the agency admin before it influences agent behavior

Standard public legal forms (Form F, compromis de vente, rogito) are the canonical reference — we point at them, we never redefine them.

This document is the durable contract. Every future compliance ticket follows this posture.

Why this exists

When HouseZoom flags "you cannot share this listing — DPE diagnostic missing," we are making a legal-adjacent claim. If that claim is wrong (rule outdated, edge case missed, regional exception ignored) and the agent acts on it, three risks materialize:

RiskExampleOur posture
Liability claimAgent sued for proceeding under stale rule we surfaced as currentCite source + disclaimer + admin acceptance log
Reputational"HouseZoom told me it was fine"Disclaimer language is permanent in UI, not just onboarding
RegulatoryA regulator views us as practicing lawWe provide tooling and references, not interpretations

We are not a legal authority. We are not a substitute for a notaire, lawyer, or qualified compliance officer. The system codifies publicly available rules from named authorities; the agency remains the responsible party.

2. Affirmative admin acceptance

A rule that has been ingested into compliance_rules is not yet active for an agency until the agency admin has reviewed and accepted it.

Rule lifecycle states for an agency:

ingested → pending_review → accepted → active
                         ↘ rejected   → inactive (audit log only)
                         ↘ overridden → active with org-specific severity/conditions

Acceptance is recorded in a new join table compliance_rule_org_acceptance:

FieldNotes
org_idFK organizations
rule_idFK compliance_rules
statepending_review / accepted / rejected / overridden
accepted_byFK auth.users (admin)
accepted_attimestamptz
acceptance_versionsnapshot of compliance_rules.version at time of acceptance
notesoptional admin reason (especially for rejected / overridden)
ip_addressfor audit defensibility

When compliance_rules.version increments materially (significant change, not typo fixes), the join row's state flips back to pending_review and the admin is re-prompted.

Default behavior for newly-ingested rules (TBD pending Will confirmation):

  • Recommended: Fire as info-only (no severity, just citation) until accepted, then promote to declared severity.
  • Alternative: Invisible until accepted (riskier — agents miss real obligations).

3. Editable ruleset (admin can review, accept, reject, override)

Admin has a dedicated UI surface for reviewing the ruleset:

  • /admin/compliance — existing surface, extended with acceptance state column.
  • /admin/compliance/review — new dedicated review queue showing all pending_review rules with side-by-side: [rule text] | [source] | [our interpretation] | [accept/reject/override actions].
  • Bulk operations: "Accept all pending FR rules" with explicit per-country acknowledgment.
  • Rejection requires a reason (free text). Override requires reason + new severity/conditions.

Anchor principle: standard public legal forms

The system points at canonical forms, it does not encode their content.

Rationale: forms change. Authorities update them. Replicating them creates a maintenance burden and a liability surface (we may surface a stale version). Pointing at the authoritative source means the agent always sees the current form.

When an agent needs to fill a form, they download from the authority's URL or use HouseZoom's existing e-signature flow with the latest authority-published template — the system never claims to be the source of truth for the form's content.

Phase 7 — Legal forms library (BL-138)

Weekly job flags rules where source_verified_at is older than 6 months (configurable per-authority — fee tables get 90-day SLA, civil code articles get 365-day SLA). Surfaces in admin dashboard. Also flagged: rules whose source_url returns 404 or material content drift (heuristic).

  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  rule_id uuid NOT NULL REFERENCES compliance_rules(id) ON DELETE CASCADE,
  state text NOT NULL CHECK (state IN ('pending_review','accepted','rejected','overridden')),
  accepted_by uuid REFERENCES auth.users(id),

  override_severity text,         -- when state='overridden'

  notes text,
  ip_address inet,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now(),
  UNIQUE (org_id, rule_id)
);

Compliance engine integration

complianceEngine.evaluateCompliance() extended to:

  1. Load rules filtered by org's acceptance state (only accepted and overridden rules fire at declared severity; pending_review rules fire as info-only; rejected rules don't fire).
  2. Filter by valid_from &lt;= NOW() AND (valid_to IS NULL OR valid_to &gt;= NOW()).
  3. Return EvaluationResult items including source_url, source_citation, source_authority, source_verified_at for UI rendering.

Out of scope (explicit)

The following are deliberately not part of this architecture:

  • Real-time legal advice. We do not interpret edge cases beyond the scope of ingested rules.
  • Cross-border transactions. Multi-country deals (e.g. French buyer of Swiss property) require professional advisors. The system can flag relevant rules from each market but does not synthesize cross-border guidance.
  • Tax optimization. We surface tax obligations as they exist in the law; we do not advise on minimization strategies.
  • AI-generated legal text. No generative-AI output that paraphrases, restates, or interprets law goes to the agent without source attribution and disclaimer rendered alongside.
  • Form generation. We point at official forms; we do not generate facsimiles. Existing e-signature flow uses authority-published templates only.
