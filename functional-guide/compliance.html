Functional Guide — Compliance

0. How to use this guide

This guide is the single source of truth for how compliance works in HouseZoom today. It is written for non-engineers (agents, agency admins, support, sales, investors) as well as engineers needing quick reference.

If you ask "what happens when X?" and the answer isn't in this guide, that's a documentation gap. Add a row to §6b (Edge Case Catalog) when you discover one. The catalog grows by discovery, not by speculation.

Maintenance rule: any session that ships user-visible behavior change updates this file. See Standing Obligations.

1. What it does (plain English)

HouseZoom's compliance engine tells agents — at a glance and in real time — whether a deal is safe to keep moving or whether it's missing something the law requires. Every market HouseZoom operates in (Switzerland, France, UAE, Italy) has its own rules: cantonal Lex Koller in Switzerland, Loi Alur / Hoguet in France, DLD / RERA in the UAE, mediation laws in Italy. Instead of asking agents to memorize the rules, HouseZoom encodes them into the workflow and continuously evaluates each deal against them. The result is a single, colour-coded signal — green, yellow, red — visible on every screen where deals appear.

In short: agents focus on the deal; HouseZoom watches the rulebook.

2. Where you see it

The compliance status appears as a coloured badge on three screens:

SurfaceWhat rendersTicket
Kanban board (/admin/deals)Small badge on each deal card next to the deal typeBG-91 (this session)
Deal detail page (/admin/deals/[id])Larger badge near the deal headerBG-115

Badge colour vocabulary is consistent across all three surfaces:

  • 🟢 OK (green) — nothing blocking. Deal is moving as expected.
  • 🟡 WARN (yellow) — something needs attention soon, but the deal isn't blocked yet.
  • 🔴 BLOCK (red) — the deal cannot legally advance until the issue is resolved. (Note: enforcement is visual today; server-side enforcement tracked as BG-90.)
  • (no badge) — this deal has no workflow attached (e.g. legacy or UAE deals predating template assignment). Compliance is not being evaluated.

3. How it works (user-facing flow)

When an agent creates or works on a deal, HouseZoom checks the deal against the rules attached to the agent's market. The rules live inside the workflow template assigned to the deal (one template per canton in Switzerland, one per region elsewhere). Each step of the workflow can be tagged with a compliance category — for example "this step requires KYC complete" or "this step requires Lex Koller authorization before continuing."

As the agent moves the deal through stages, fills in fields, uploads documents, or adds parties, the compliance engine re-evaluates and updates the badge. The agent does nothing special — the signal just appears.

Example walkthrough (live in Zürich Premium Realty seed):

  • P1 Kilchberg Villa — Brunner → 🟢 OK — KYC complete, parties confirmed, no Lex Koller trigger, deal can move freely.
  • P5 Bellerive Penthouse — Rinaldi → 🟡 WARN — likely a soft warning (missing GEAK or near-deadline document); deal can still move, but the agent should resolve it before the next stage.
  • P4 Zollikon Townhouse — Sutherland (person-abroad) → 🔴 BLOCK — Sutherland is a buyer abroad; Swiss Lex Koller requires cantonal authorization before the sale to a foreign buyer can proceed. The badge tells the agent to stop and resolve before doing anything else.
  • Aktiv – Manuel Kaufmann → no badge — this is a lead-stage deal not yet attached to a workflow; nothing to evaluate.

3a. Capturing Lex Koller residency status on deal participants

Two P1 UI bugs (acceptance-panel refresh + recompute auth) that silently swallowed the success signal were closed in D-368; the acceptance UI + Save → recompute UI now refresh correctly. The original engine-fix TD filed yesterday is invalidated and a candidate for reformulation/closure.

Switzerland's Lex Koller restricts foreign residents' acquisition of Swiss real estate. To evaluate whether a sale is exempt, requires authorization, or is outright blocked, the engine needs to know each buyer's residency status. That's now captured directly on the deal participant.

Where the field appears

On the deal detail panel for any Swiss deal (deals with market_country_code = 'CH'), each buyer and seller participant row now shows a "Lex Koller residency status" dropdown. The field is hidden for non-CH deals (FR / IT / AE) and for non-buyer/seller roles (buyer_agent, listing_agent, notary, bank, etc.). It also appears in the "Add participant" form on CH deals when the chosen role is buyer or seller.

The 4 values, in plain language

ValueWhat it means
Person abroad (foreign resident)Buyer/seller is resident outside Switzerland. Triggers federal Bewilligung path.
Swiss residentResident in Switzerland without permit requirement.
Swiss resident with permitResident in Switzerland holding the required permit (e.g. C-Bewilligung, B-EU/EFTA).
UnknownAgent has asked, answer not yet known. Same engine treatment as "— not specified —" today, but the intent is different and preserved for future audit/reporting.
— not specified —Field has never been touched. Distinct from Unknown (which means asked-but-not-answered).

Saving

Each existing buyer/seller row has its own Save ("Enregistrer") button next to the dropdown — scoped LK-only, no other field on the participant is touched. The button label cycles Save → Saving… → Save during the round-trip. On success, a fresh compliance evaluation runs for the deal (POST /api/deals/[id]/recompute-compliance), and the deal reloads so the latest evaluation surfaces. The Save button stays disabled until the dropdown value actually differs from what's stored.

What the engine evaluates

Three federal Lex Koller rules read this status (data-only migration 20260528_td_crm_ui_lex_koller_federal_conditions.sql):

Rule codeBehavior when buyer is person_abroad
CH-SALE-LEX-KOLLER-AUTHORIZATION-001 (BewG Art. 15)Designed to block — cantonal Bewilligung required before notarisation.
CH-SALE-LEX-KOLLER-EXEMPTION-001 (BewG Art. 7)Flag/warning — document the exemption.
CH-SALE-LEX-KOLLER-STATUS-DECLARATION-001 (BewG Art. 2)Flag/warning — Form 3 declaration required.

When a buyer is not person_abroad, all three resolve as non-blocking warnings via the rule's when_unmatched branch. When residency is unknown, NULL, or "— not specified —", the rules resolve via when_unevaluable (same: warn, no block).

Current limitation — honest disclosure

The data capture, server persistence, and compliance recompute pipeline are in place. The engine reads buyer residency correctly and produces fresh evaluation records on each Save (visible in compliance_evaluations, with context_snapshot.buyer_lex_koller_residency_statuses populated as expected). The final step — engine-side application of when_matched.blocking: true to actively block stage advancement — is not yet active. AUTHORIZATION warnings surface on the panel, but no hard block fires. Tracked as TD-COMPLIANCE-ENGINE-CONDITIONAL-RESOLUTION (🔴 P1).

Until that ships, agents working CH deals with foreign buyers should treat the AUTHORIZATION warning as actionable, not optional, and pursue the federal Bewilligung application before stage advancement.

Re-hydration caveat (BG-118, 🟡 P2)

After clicking Save, the dropdown may visually reset to empty even when the saved value is unknown — the database has the correct value, but the panel's local state doesn't re-sync on the next interaction. Avoid clicking Save a second time without first re-opening the deal detail to verify. Tracked as BG-118 — UI re-hydration after Save.

Roles covered

Buyer + seller participants only. Other roles (buyer_agent, listing_agent, notary, bank) do not show the field today. Future tickets may extend if a rule requires it (e.g. legal-entity buyers — TD-COMPLIANCE-LEX-KOLLER-LEGAL-ENTITIES).

Non-CH deals

The field is hidden on FR / IT / AE deals. Lex Koller is exclusively Swiss federal law; no equivalent capture exists for other markets today.

4. What drives it (data sources)

The badge data comes from one place: a field called compliance_status on the deal's attached workflow instance (one workflow instance per deal that's running on a template).

  • The field has exactly three possible values: ok, warn, block.
  • A deal with no workflow instance attached returns no value — the badge simply doesn't render.
  • The agent never edits this field directly. It's set by the compliance engine when: - The deal is created against a market template - A workflow stage advances - A document is uploaded, signed, or expires - A party is added, removed, or has KYC status changed - A nightly re-evaluation pass runs (planned for BL-153a)

The badge UI itself is purely visual — it reads compliance_status and picks the colour. No logic on the kanban card or the detail page makes its own compliance judgement.

4a. Who can author / edit rules

Added 2026-05-29 (D-372 / D-373).

🟢 Update — 2026-05-29 (D-372 / D-373): authoring trust boundary tightened. Statutory rules (base / country / region layers — shared across all agencies via org_id IS NULL) are now writable only by a platform admin via the new is_platform_admin() gate. Org admins are restricted to their own org's company-layer rules and can no longer write statutory_lock=true. The cross-tenant write hole that previously let any agency admin edit shared statutory rules (BG-122) was closed in the same session. Verified live both directions: agency admin → 403 on statutory writes; platform admin → 201/200/200 on statutory insert/update/delete. The platform-admin role is currently a database-level security boundary only — the authoring UI (/superadmin/compliance) is planned (BL-162 Unit 3, depends on BL-163) but not yet built.

Compliance rules sit on a 4-layer hierarchy. Who is allowed to author or edit a rule depends entirely on which layer it's on:

Rule layerScopeWho can write
base / country / region (statutory)shared across all orgs (org_id IS NULL)Platform admin only
companyone agency (org_id = that org)That org's owner or admin only

The split exists because statutory rules encode law — editing one changes the rule for every agency on HouseZoom. Company rules are private to an agency. A platform admin is a HouseZoom-internal role (legal/ops, cross-organization), not an agency role; see platform-admin.md for the full picture (who, why, how the role is granted, what's not yet built).

A few practical corollaries:

  • An agency owner or admin attempting a statutory write fails at the database with 42501 (row-level security denial). There's no app surface that even tries.
  • statutory_lock = true may be set only by a platform admin. The company-layer write policies explicitly disallow it.
  • Reading statutory rules is open to all agencies (intentional — source-attribution UX, agent transparency). Write scope is independent of read scope.
  • Today there is exactly one platform admin (the founder). Statutory rules are still authored via SQL / seed scripts / service_role — the /superadmin/compliance screen is not yet built.

5. Markets & locale variance

Compliance rules vary by market, but the badge surface is uniform — same colours, same placement, same meaning everywhere. What differs is what causes a badge to go yellow or red.

MarketTriggers that move a badge to WARN or BLOCK
Switzerland (CH)Lex Koller person-abroad authorization; GEAK energy certificate missing; cantonal-specific document deadlines; notary appointment confirmation; BL-129 stage-category gates
France (FR)Loi Alur diagnostic bundle (DPE, ERP, etc.); Hoguet card validity for mandate; mandate notice period; compromis-to-acte timing

When the agent's locale is French, the badge labels currently still render as raw English (OK / WARN / BLOCK) — this is intentional consistency with the other two badge surfaces. Full localization (with compliance.status.ok, .warn, .block keys) is planned as a single-pass update across all three surfaces (filed as TD-COMPLIANCE-BADGE-I18N 2026-05-27).

6. Known limitations & gotchas

These are intentional behaviors or known shortcomings of the current implementation. They're stable enough to document upfront. Use this section to set expectations with new users; use §6b for cases discovered in the wild.

  • No workflow attached → no badge. Correct behavior. Common for legacy deals migrated before template assignment, lead-stage deals not yet promoted, and UAE deals predating template rollout.
  • Workflow attached but evaluation hasn't run yet → defaults to ok. The database default. Agents will see green until the first evaluation finds something. This is an acceptable optimistic default; a future enhancement may render a neutral "pending" badge instead (not yet scoped).
  • Block state never enforces server-side today. A red badge tells agents "stop" but doesn't yet prevent stage advance at the server. Server-side enforcement is tracked as BG-90.
  • Workflow templates without compliance categories on stages → behaves as ok. Stages without a canonical category contribute nothing to evaluation. Currently relevant for any non-CH template until BL-129-style categorization is extended market by market.
  • Badge labels render in raw uppercase English even on FR/DE/IT locales. Intentional cross-surface consistency until TD-COMPLIANCE-BADGE-I18N ships.

8. Glossary (jargon-free)

  • Workflow template — a market-specific recipe of stages, document requirements, and rules. One per canton in Switzerland; one per region elsewhere.
  • Workflow instance — the live copy of a template attached to a specific deal. Holds the deal's current stage, document statuses, and compliance state.
  • Stage — a discrete step in the deal lifecycle (e.g. Qualification, Mandat, Promesse, Acte authentique).
  • Canonical category — the compliance bucket a stage belongs to (e.g. pre_offer, transfer). Determines which rules apply at which stage.
  • Compliance engine — the code that reads the deal + workflow instance + rules and outputs ok / warn / block. Today: synchronous on deal mutations. Future: also nightly background pass.
  • Lex Koller — Swiss federal law restricting real estate sales to non-residents; requires cantonal authorization for certain buyer profiles.
  • Loi Alur / Hoguet — French laws governing real estate transactions and the agent profession (mandate, diagnostics, agent licensing).
  • DLD / RERA — Dubai Land Department / Real Estate Regulatory Agency. UAE regulators.
  • KYC — Know Your Customer. The identity-verification process for parties in a transaction. Mandatory under AML laws.
  • AML — Anti-Money Laundering. The regulatory framework KYC sits inside.
