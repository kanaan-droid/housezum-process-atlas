Functional Guide — Platform Admin (Superadmin)

Status: Living document. Updated end-of-session whenever platform-admin behavior or scope changes. Version: v0.1 Last updated: 2026-05-29 (BL-162 Unit 1 / D-373) Maintainer: Will + Claude

0. How to use this guide

This guide is the single source of truth for what a HouseZoom platform admin is, what it can do today, and what it deliberately can't do. It's written for non-engineers (support, sales, investors) as well as engineers needing quick reference.

Pair this guide with compliance.md. The two are tightly coupled: the platform-admin role exists because of how compliance rules are layered, and changing one without the other is a recipe for drift.

Maintenance rule: any session that ships user-visible behavior change to the platform-admin role updates this file. See Standing Obligations.

1. What it is (plain English)

A platform admin (also called superadmin in some specs) is a HouseZoom-internal role for legal and operations staff. It sits above the per-agency role hierarchy and is cross-organization — a platform admin is not a member of any single agency. There is exactly one platform admin today: the founder.

The role exists for exactly one purpose at this time: authoring the statutory compliance rules (the base, country, and region layers) that are shared across every agency on the platform.

If an agency owner is the "CEO of one agency's account on HouseZoom," a platform admin is "the person at HouseZoom who keeps the rulebook for all agencies up to date with the law."

2. Why it exists

HouseZoom's compliance engine works on a 4-layer rule hierarchy (see compliance.md for the full picture):

base       → applies to every market
country    → applies to all orgs in a country (e.g. CH, FR, IT, AE)
region     → applies to all orgs in a region/canton (e.g. CH-ZH)
company    → applies only to one agency

The first three layers are statutory — they encode law. Their rows live in compliance_rules with org_id IS NULL, meaning they're not owned by any agency. Editing one of those rows changes the rule for every agency on the platform. So no org-level admin — even an agency owner — may write them. Only a platform admin can.

The fourth layer (company) is the opposite: each row is owned by exactly one agency, scoped via org_id. Agency owners and admins author their own company rules within their own org. They cannot touch any other agency's company rules, and they cannot touch any statutory rule.

This split is what the platform-admin role enforces.

3. Who can author what

This is the rule readers most need to find quickly.

Rule layerScopeWho can write
base / country / region (statutory)shared across all orgs (org_id IS NULL)Platform admin only
companyone agency (org_id = that org)That org's owner or admin only

A few corollaries that matter in practice:

  • An agency owner or admin cannot create, edit, or delete a statutory rule. Attempting it fails at the database with 42501 (row-level security denial).
  • An agency owner or admin cannot write a row with statutory_lock = true — the gate is enforced even on the company-layer write path.
  • A platform admin can set statutory_lock = true on statutory rules they author. This is the marker used elsewhere (engine, future UI) to flag a rule as legally non-overridable.
  • An agency admin from one org cannot write any rule belonging to another agency. They can only write their own org's company rules.
  • The database read view is wider than the write view: every agency can read the statutory rules that apply to them (this is intentional — source-attribution UX, agent transparency). Read scope and write scope are independent.

4. How someone becomes a platform admin

Provisioning is manual SQL only, by Will or ops, inserting a row into the platform_admins table.

There is deliberately:

  • No sign-up flow. No one self-promotes.
  • No invitation UI. The system has no screen for granting this privilege.
  • No org admin path. Org owners and admins cannot grant it; the role is outside the per-org hierarchy entirely.
  • No public discovery. The platform_admins table is invisible to every authenticated app user (it's row-level-security-locked with no policies — the table simply doesn't appear in SELECTs from any normal client). Only the database's service_role and the is_platform_admin() function can see it.

The intent of these constraints is simple: a privilege that affects every tenant on the platform should be ungrantable through the platform's own UI.

5. Current state (2026-05-29)

Live and verified:

  • The platform_admins table exists and is provisioned with the founder, will@housezoom.ai. (Granted 2026-05-29 22:15 UTC.)
  • The database function is_platform_admin() returns true for that user and false for everyone else.
  • The three statutory write policies on compliance_rules are active. An org admin attempting a statutory write is denied; a platform admin attempting one succeeds. Both directions verified live the day of ship.

Not yet built:

  • No authoring UI exists. A platform admin cannot today open a screen and edit a statutory rule. Statutory rules are still created and edited only via SQL (seed scripts, migrations, manual statements). The DB enforces who can write — but there is no app surface that exposes the privilege.
  • The planned screen is /superadmin/compliance, gated on is_platform_admin(). It depends on the BL-163 rule-authoring form. Both are in the backlog (see §7).

Until the UI lands, the practical impact of the role is purely a security boundary: it lets HouseZoom apply legal updates via the database with a single authorized actor, and it blocks every agency from accidentally (or deliberately) editing shared statutory rules.

6. Known limitations & gotchas

  • No app surface today. Repeated for emphasis: "platform admin" is not a screen. If someone asks "where do I log in as platform admin?" the honest answer is: you already log in as yourself; the privilege is attached to your user, the role is invisible in the UI today, and the only place you can use it is via direct DB writes.
  • Single platform admin. Only the founder is provisioned. There's no operational reason this couldn't grow — the table is built for N rows — but it's intentionally kept at one until there's a real need.
  • Revocation is also manual SQL. Removing a platform admin is a DELETE FROM platform_admins WHERE user_id = … by ops. No UI, no audit trail beyond the granted_by / granted_at columns on the original grant.
  • Audit of statutory writes is not yet append-only. Until BL-162 Unit 2 ships the API route, statutory writes go through ad-hoc SQL. An audit log of statutory edits is part of Unit 2's scope, not this commit's.

8. Glossary (jargon-free)

  • Platform admin / superadmin — a HouseZoom-internal role authorized to author statutory compliance rules across all agencies. Distinct from any agency's owner or admin. Stored in the platform_admins table.
  • Statutory rule — a compliance rule encoding law, applied to all agencies (the base, country, and region layers). Marked with org_id IS NULL.
  • Company rule — a compliance rule specific to one agency (the company layer). Marked with org_id = &lt;that agency's id&gt;.
  • statutory_lock — a boolean flag on a compliance rule. When true, the rule is treated as legally non-overridable. Only a platform admin can set it.
  • Row-level security (RLS) — a Postgres feature that constrains which rows a given user can read or write at the database level, independent of application code. The platform-admin trust boundary is enforced via RLS.
  • is_platform_admin() — a SQL function that returns true if the current user has a row in platform_admins, false otherwise. Used inside RLS policies to gate statutory writes.
  • service_role — the privileged Supabase database role that bypasses all RLS. Used by server-side API routes for trusted operations. Seed scripts and migrations run as this role.
