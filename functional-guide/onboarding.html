Functional Guide — Self-Serve Onboarding & Auth

Status: Living document. Updated end-of-session whenever signup, auth, or onboarding behavior changes. Version: v0.1 Last updated: 2026-06-06 (initial — BL-165/168 + BG-123→128 founder-flow chain) Maintainer: Will + Claude

0. How to use this guide

This guide is the single source of truth for how a new user gets into HouseZoom today: from the signup form, through email confirmation and auto-login, to the onboarding choice (solo vs. agency), agency creation, profile provisioning, and the first landing screen inside the app. It is written for non-engineers (support, sales, investors) as well as engineers needing quick reference.

Pair this guide with platform-admin.md and compliance.md. This guide covers how an agency user self-onboards. The platform-admin role is provisioned by a completely different (manual, SQL-only) path — see platform-admin.md §4. There is deliberately no overlap: nothing in self-serve onboarding can grant platform-admin.

If you ask "what happens when X during signup/onboarding?" and the answer isn't here, that's a documentation gap. Add a row to §8 (Edge Case Catalog) when you discover one.

Maintenance rule: any session that ships user-visible behavior change to signup, the auth callback, or onboarding updates this file. See Standing Obligations.

1. What it is (plain English)

Self-serve onboarding is how a brand-new person turns "I have an email address" into "I'm working inside a HouseZoom agency account" without anyone at HouseZoom touching a database.

The journey has five stages:

1. Sign up        →  email + password, then "check your email"
2. Confirm        →  click the email link → /auth/callback → logged in
3. Choose         →  "Just me" (solo)  or  "I'm setting up for my agency"
4. Create         →  (agency path) name + country + region → org + market config
5. Land           →  profile provisioned, dropped into the app (no manual refresh)

The person who completes the agency path is the founder of that agency: the first user, the owner, auto-approved. Everyone they invite afterwards goes through the separate invite flow (not covered here — see the invite-signup path).

2. Why it works the way it does

Two hard constraints shape this whole flow. Knowing them explains every design decision below.

A. Sessions live in the browser, not in a cookie. HouseZoom uses Supabase's implicit auth flow: after a successful login the access/refresh tokens arrive in the URL hash and are persisted to localStorage (storageKey: 'housezoom-auth'), not to a cookie. The practical consequence: the server cannot see who you are on a normal navigation. Any routing decision that depends on "am I logged in / approved / a member of an org" has to happen client-side, after the browser has read its own session. This is why the auth callback is a client page that waits for the session before routing, and why post-onboarding navigation is a soft client-side router.replace, not a server redirect.

B. A fresh user cannot create their own profile. The agent_profiles table is row-level-security-locked. The self-insert policy (agent_profiles_self_insert) requires WITH CHECK (auth.uid() = user_id AND org_id = get_user_org_id()) — i.e. you may only insert your own profile row if it already points at the org you already belong to. A brand-new user belongs to no org yet, so that check can never be satisfied from the browser. (The legacy database trigger that used to auto-create a profile row, handle_new_user, was dropped in migration 003_drop_legacy_users.sql.) The consequence: profile creation must run server-side with the privileged service_role, which bypasses RLS. That's exactly what ensureProfile does (§6).

Everything else in this guide is downstream of those two facts.

3. Stage 1 — Signup (/auth/signup)

The user enters an email and password. On submit:

  • The page signs out first. If someone is already authenticated and lands on signup, we clear that session before creating the new one, so a stale session can't leak into the new account. (There's also a mount guard: an already-authenticated user arriving at /auth/signup is handled rather than silently double-signed-up.)
  • We call Supabase signup with emailRedirectTo: ${origin}/auth/callback — this is what makes the confirmation email's link point back into our app at the callback route (§4), regardless of which environment (preview/prod/local) the user signed up from.
  • We then read the signup outcome (resolveSignupOutcome) and branch on it:
OutcomeWhenWhat the user sees
check-emailSupabase returns no session (email confirmation required)A "check your email" confirmation state (confirmSent). The user is told to click the link in their inbox. Nothing else happens until they do.
proceedA session already exists (confirmation not required in this environment)Straight through to routing.
noneNo user came back at allTreated as an error state, not a silent success.

The check-email state is the common one in production: signups require email confirmation, so session is null immediately after submit. The page must handle that null gracefully (it does) rather than assuming a logged-in user.

Why signOut-first matters: without it, a returning visitor with a half-alive session could submit signup and end up with the old identity attached to the new form action — a confusing cross-account state. Clearing first makes signup deterministic.

4. Stage 2 — Email confirm → auto-login (/auth/callback)

The confirmation email link lands the user on /auth/callback, a client page (it has to be — see §2A). Its only job is: establish the session, then route the user to the right next screen.

It waits for the session to materialize rather than assuming it's there:

  • It subscribes to onAuthStateChange and calls getSession() — whichever resolves the session first wins. (Implicit-flow tokens come in via the URL hash and are parsed asynchronously, so the session isn't guaranteed to exist on first render.)
  • If no session appears within an 8-second fallback, it gives up and sends the user to /auth/login rather than hanging forever.
  • Once the session exists, it routes via resolvePostLoginRoute using a soft router.replace(target) — a client navigation, never a server redirect (again, §2A: the server can't see the session).

resolvePostLoginRoute — the routing brain

This one pure function decides where any logged-in user goes, and it's reused beyond the callback (onboarding success, login). Inputs: { activeInvite, approved, hasActiveMembership, role, isDesktop }.

Condition (checked in order)DestinationMeaning
activeInvite present/invite/${token}User has a pending invite → finish accepting it.
approved and admin/owner role and desktop/adminAn approved owner on desktop goes to the admin console.
approved/dashboardAn approved user goes to the app.
no active membership/onboardingFresh founder — belongs to no org yet → go build one.
otherwise/pending-approvalMember of an org but not yet approved → wait.

For a brand-new founder who just confirmed their email, the path is: no membership → /onboarding.

5. Stage 3 — The onboarding choice (/onboarding)

The onboarding screen asks one question with two answers:

ChoiceWhat it meansWhere it goes
"Just me"A solo agent, no agency setup.Solo path → /dashboard (no org is created; the user works as an individual).
"I'm setting up for my agency"This person is the founder of an agency on HouseZoom.Agency-creation path (§6).

Solo routing is deliberate. A "just me" user lands on /dashboard, not /admin. (This was an explicit correction during BG-126: solo users are not org admins and have no admin console to see.)

Both success handlers run through finishOnboarding (§7), which guarantees the app's org state is refreshed before the user is navigated anywhere.

6. Stage 4 — Agency creation (POST /api/onboarding)

When the founder chooses the agency path and fills the form, the server route creates everything an agency needs in one shot. Inputs from the form: org name, country, and a region selection.

6.1 Country & region (data-driven)

  • Country is one of the five live markets: CH, FR, IT, DE, AE.
  • The region picker is data-driven from src/lib/constants/market-regions.ts — the UI renders whatever regions that config declares for the chosen country. The rules per market:
CountryRegionsRequired?
CH (Switzerland)26 cantonsRequired — a Swiss org must pick its canton (reuses the canonical SWISS_CANTONS list with authored labels).
AE (UAE)7 emiratesOptional
DE (Germany)16 LänderOptional
IT (Italy)20 regionsOptional
FR (France)—None (no region layer)
  • Timezone is derived from the country/region rather than asked.

6.2 Market configuration (from market_defaults)

The org's operating defaults are looked up from the market_defaults table by country (via src/lib/marketDefaults.ts) and written onto the org. The three configured dimensions are:

  • currency (e.g. CHF, EUR, AED)
  • measurement_unit — metric for CH/FR/IT/DE; AE = sqft (the one imperial market)
  • default_locale — the org's starting language

The locale resolver maps the chosen market to a sensible default language — e.g. a Geneva (canton GE) org resolves to fr, not German — so a founder isn't dropped into the wrong language because of a country-level default.

6.3 Region dual-write + the primary market row

  • The selected region is written as region_code and, for Switzerland, also to the canton column (a dual-write so both the generic region model and the Swiss-specific canton model stay consistent — BL-165).
  • Exactly one primary organization_markets row is created for the org's home market. (The schema supports an org operating in multiple markets later; onboarding seeds the single primary one.)

6.4 Profile provisioning (the critical step)

This is where §2B is paid off. The route calls ensureProfile with the service_role client (bypassing RLS, because the browser physically cannot do this insert):

  • It upserts the agent_profiles row (onConflict: 'user_id'), so the call is idempotent — a retry can't create a duplicate.
  • The founder is auto-approved (approved: true). The person who creates the agency doesn't wait in a pending queue — they're the owner.
  • operating_country is inherited from the org's country (the ISO code, e.g. CH). It is never hardcoded — see the BG-125/127 history in §9 for why a stray 'UAE' default was a live blocker.
  • phone_country_code (the phone dial code) is inherited from the org country via resolveDialCode — CH→+41, FR→+33, IT→+39, DE→+49, AE→+971. It is never silently +1.
  • If profile provisioning fails, the route treats it as fatal (HTTP 500) rather than swallowing the error and leaving the user in a half-created state (BG-125). A founder with an org but no profile is a broken account, so we'd rather fail loudly.

7. Stage 5 — Landing in the app (finishOnboarding + OrgProvider.refresh)

The last hazard is a navigation race: the app's org state (OrgProvider) is loaded once on mount and on auth-state change — it does not automatically notice that the user just created a brand-new membership. If we navigated immediately after creating the org, the app would still believe the user has no org and bounce them right back to /onboarding (the BG-128 bug).

The fix is finishOnboarding(refresh, navigate, target), which enforces the order:

await refresh()      // re-resolve org/membership state first
navigate(target)     // only then move the user
  • refresh is OrgProvider.refresh() — exposed on the org context specifically so onboarding can force a re-resolve after creating a membership.
  • Only after the refresh resolves does the user get navigated, so they land in the app with correct state and no manual page refresh required.
  • Final destination: an owner on desktop → /admin; everyone else → /dashboard (computed via resolvePostLoginRoute, the same brain from §4).

9. Known limitations & follow-ups (do not assume fixed)

  • TD-DATA-COUNTRY-CODE-COLLISION 🟡 — agent_profiles has two country-ish columns that mean different things: operating_country (ISO-2, the market the agent works in, ISO-checked) and country_code/phone_country_code (the phone dial prefix, e.g. +41, with no ISO check and a legacy '+1' default). The names invite confusion. Tracked rename pending.
  • TD-ONBOARDING-PROFILE-ROLLBACK 🟡 — agency creation writes several rows (org, organization_markets, profile) without a single enclosing transaction. A partial failure mid-sequence can leave an inconsistent set. Profile failure is at least surfaced as a fatal 500 (BG-125), but a true all-or-nothing rollback is not yet in place.
  • TD-TEST-DB-INTEGRATION-HARNESS 🟡 — the onboarding/auth tests exercise pure helpers (resolvePostLoginRoute, resolveSignupOutcome, ensureProfile payload shape, resolveDialCode, resolveOperatingCountry, finishOnboarding ordering) rather than real DB calls, because the shared mocked Supabase client can't reproduce RLS, CHECK constraints, or .insert().select().single() chaining. Constraint-level regressions (like the operating_country default) therefore cannot be caught by unit tests today — they need a real-DB integration harness.
  • next lint removed under Next 16 — the repo's "lint": "next lint" script is dead (Next 16.1.6 removed the next lint subcommand). Lint runs via npx eslint directly until the script is repointed to eslint .. Not specific to onboarding, flagged here because it surfaced during this work.

11. Glossary (jargon-free)

  • Founder — the first user of an agency; the one who completes the agency-creation path. Owner role, auto-approved.
  • Solo / "just me" — a user who onboards without creating an agency; works as an individual, lands on /dashboard.
  • Implicit auth flow — the Supabase login mode HouseZoom uses: tokens arrive in the URL hash and live in localStorage, not a cookie. The server can't read the session, so login-dependent routing happens in the browser.
  • /auth/callback — the client page the email-confirmation link lands on; it waits for the session, then routes via resolvePostLoginRoute.
  • resolvePostLoginRoute — the pure function that decides where a logged-in user goes (invite / admin / dashboard / onboarding / pending-approval).
  • ensureProfile — the server-side, service_role upsert that creates the agent_profiles row. The only way a profile can be created, because RLS blocks the browser from doing it.
  • service_role — the privileged Supabase database role that bypasses row-level security. Used by trusted server routes (and seed scripts/migrations).
  • organization_markets — the table holding an org's market(s). Onboarding seeds exactly one primary row for the home market.
  • market_defaults — the table mapping a country to its currency / measurement unit / default locale; read during agency creation.
  • resolveDialCode — maps an org country to its phone dial prefix (CH→+41 … AE→+971), never +1.
  • OrgProvider.refresh() — re-resolves the user's org/membership state on demand; awaited after agency creation so the app sees the new membership before navigating.
