1. Overview

1.1 What's seeded

The HouseZoom production database hosts two operational concerns in the same tables:

  1. Real platform data — actual users, real orgs, anything created via the app UI by Will or future paying customers.
  2. Seeded demo/staging data — fixture data populated by scripts to make the product demoable, testable, and visually realistic across all target markets (UAE, FR, DE-CH, IT).

Seeded data is always discoverable via specific markers. There is no "seeded data is also production data" gray zone — every seeded row carries an identifier explained in § 1.3.

1.3 The marker convention

Every seeded row from a script using marker discipline carries a string of the form [seed:&lt;scope&gt;-v&lt;n&gt;] appended to its notes field.

Why this pattern:

  • Schema-clean — no dedicated is_seed column, no migrations needed.
  • Visible — markers show up in UI (mildly ugly but signals "this is demo data" to anyone inspecting).
  • Surgical rollback — DELETE WHERE notes LIKE '%[seed:&lt;scope&gt;-v&lt;n&gt;]%' only touches seeded rows, never real data.
  • Re-runnable — seed scripts check for existing markers before running; abort with --force instructions if found.

Gulf Horizon caveat: the original UAE seed shipped before this convention. seed-dubai.mjs is idempotent (re-run safe via deterministic UUIDs) but rolling back individual rows requires the FK-ordered wipe SQL in scripts/SEED-INSTRUCTIONS.md. Future re-seed of UAE should retrofit the marker pattern.

  o.name,

  (SELECT COUNT(*) FROM contacts WHERE org_id = o.id AND notes LIKE '%[seed:eu-v1]%') AS contacts,
  (SELECT COUNT(*) FROM deals WHERE org_id = o.id AND notes LIKE '%[seed:eu-v1]%') AS deals,
  (SELECT COUNT(*) FROM viewings WHERE org_id = o.id AND notes LIKE '%[seed:eu-v1]%') AS viewings,
  (SELECT COUNT(*) FROM leads WHERE org_id = o.id AND notes LIKE '%[seed:eu-v1]%') AS leads
FROM organizations o
WHERE o.id IN (

  '0bdbef88-937b-41b8-8885-a2eedf4de323'

ORDER BY o.name;

6.2 Future-proofing

Any future migration that tightens nullability on a currently-nullable legacy column (e.g., making viewings.user_id NOT NULL) will surface at runtime in seed-eu-locales.mjs via formatPgError with a parsed diagnostic. No silent failure path — design intent.

8. Lessons learned from BL-126 (for future seed work)

  1. Diagnostic-first. BL-126 ran SQL discovery (agents → properties → photo coverage → CRM gap) BEFORE writing the spec. This is the same workflow that worked for BG-79b. Spec quality is downstream of diagnostic quality.
  2. Spec before script. A markdown spec with counts, distribution tables, and locale data libraries took ~1 hour to write. Saved at least a day of CC iteration.
  3. DB-SCHEMA.md is a hint, not a contract. Verify CHECK constraints against pg_constraint before any insert-heavy work. § 6.1 is the canonical query.
  4. NOT NULL audits must cover the entire target table, not just the column that fired. BL-126 hit viewing_feedback.user_id NOT NULL, patched it, and immediately hit viewing_feedback.org_id NOT NULL on the next run. Two CC iterations on the same table because the first audit was column-scoped instead of table-scoped.
  5. Idempotency markers in notes are clean enough. No dedicated is_seed column needed. Trade-off: marker is visible in UI (mildly ugly). Acceptable for demo orgs.
  6. --force (rollback + re-seed) is mandatory for iterative demo prep. Append-only creates duplicates that pollute fast. --rollback then --seed two-step is too slow for tweaking. --force in one shot is the sweet spot.
  7. CC's deviations are usually correct. When CC pushes back on a spec detail, the deviation is more often a real catch than over-engineering. BL-126 examples: viewing_feedback.buyer_interest_level (correct DB column name vs spec's interest_level), monthly-rent budget shape for tenant leads, audit summary block in script header. Pattern noted in last 3+ sessions.
  8. Phase 0 advisory > programmatic pre-flight. Supabase JS doesn't expose pg_constraint (PostgREST publishes only public schema). Manual SQL block in seed script header + runtime CHECK violation parser via formatPgError was the cleanest fit.
