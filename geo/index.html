Geo & Locations — Architecture

Status: Living document. Updated when geo-resolution architecture changes. Version: v0.1 Last updated: 2026-06-08 (BL-POSTAL-TO-REGION — CH live, UI-verified) Maintainer: Will + Claude

1. Purpose

Many HouseZoom compliance rules are sub-national: Swiss rules vary by canton, German rules by Bundesland. The compliance engine matches rules on a precise region identifier. Geo-resolution is the cross-cutting capability that turns the location data an agent naturally provides into that identifier — without asking the agent to understand jurisdictions.

This is a capability, not a country feature. The mechanism is identical for every market; only the underlying data differs per country. Per-country data coverage is tracked separately under Data Environments → Geo Data Seeding.

2. The field taxonomy (foundational)

Four location fields exist on a property. Conflating them is the single most common source of confusion.

Plus, at the country level: the property's country (name, e.g. Switzerland) resolves to an ISO-2 code (CH) at call-time via legacyCountryNameToCode. (Storing ISO country_code on the property is a planned canonicalization — see §7.)

The load-bearing relationship is postal_code → region_code: the agent supplies the first, the system derives the second. region is ignored by the engine.

3. The data asset — postal_regions

A seeded reference table mapping postal codes to ISO 3166-2 regions.

ColumnMeaning
country_codeISO-2 (e.g. CH)
postal_codeFull postal code (CH 4-digit, DE 5-digit), stored as text (preserves leading zeros)
region_codeISO 3166-2 region (e.g. CH-ZH)
is_ambiguoustrue when the postal code spans more than one region; region_code holds the cross-validated primary

Primary key (country_code, postal_code) — one row per postal code, primary region only. Read-only reference data: RLS permits read for authenticated users; writes go via service role (seed scripts).

Ambiguity handling: a small number of postal codes straddle regional borders. Rather than store multiple rows, the table stores the primary region and flags the row is_ambiguous. The primary is chosen from official data and, for genuine border cases, cross-validated. Resolution never blocks a transaction; the region can always be overridden.

4. The resolver — resolvePostalToRegion

resolvePostalToRegion(countryCode, postalCode) → region_code | null

  1. Normalize the postal code (trim; pad to the country's expected width).
  2. Exact lookup against postal_regions on (country_code, postal_code).
  3. Validate the resolved code is a real subdivision for that country (guards against a bad seed row).
  4. Match → ISO region. No match → null. The resolver never guesses.

Null is a valid, safe outcome: the property keeps its postal code, no region is derived, and the deal runs on national rules only — correct degradation.

6. Extending to a new country

The capability is country-agnostic. Bringing a country online is a data task, not a code task:

  1. Source an authoritative postal→region dataset (official directory; reuse-licensed).
  2. Cross-validate genuinely ambiguous (border-straddling) codes.
  3. Seed postal_regions for that country_code (idempotent seed script; Will runs).
  4. Derivation switches on immediately — no code change.

Note: regional rule authoring (e.g. German Bundesland rules) is a separate content track. Derivation can be live for a country before that country has any regional rules to fire — the wiring is harmless ahead of the rules.

Live coverage is tracked under Data Environments → Geo Data Seeding.
