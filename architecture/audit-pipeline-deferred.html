Key reframe

The 4 tables are not competing audit tables — they have distinct purposes:

  • audit_log is the only one shaped for audit; nobody writes to it
  • activity_log is a user-facing comms timeline (read by profileService.ts)
  • activity_events is operational event stream consumed by orchestrator + UI
  • deal_stage_history is purpose-built for compliance reads + UI timelines

No consolidation needed. Original Section 3 ("keep/merge/rebuild") collapsed once evidence came in.

What needs to be built (when we come back)

Phase 1 — Event taxonomy (~30 min)

Lock the ~20 events worth logging. Three families: auth (login / role change / password / membership), tokens (mint / revoke / consume), sensitive business actions (deal delete, compliance override accept, document sign, contact merge).

Draft taxonomy table (see § Appendix below).

Phase 2 — App-side wrapper (~2-3h)

New route POST /api/audit/event with service-role write. Helper src/lib/auditLog.ts mirroring shape of existing src/lib/activityEvents.ts. Fail-open default (log error, return 200 — primary action not blocked).

Phase 3 — DB triggers (~3-4h)

Trigger function audit_log_writer() declared SECURITY DEFINER SET search_path = public, pg_temp (matches TD-AUTH-SECDEF-SEARCHPATH pattern from D-352). Per-table AFTER triggers on: deals (UPDATE, DELETE), properties (DELETE), contacts (DELETE), documents (DELETE), compliance_rules (UPDATE).

Phase 4 — Wire callers + auth events (~2-3h)

Modify ~10 existing handlers to call logAudit({...}) after their primary action: login handler, role change UI, token mint paths, compliance override accept flow, document signing completion. Auth event capture via Supabase Auth Hooks if plan tier supports; else app-side login handler wrapper.

Phase 5 — Verification window (1 week)

Audit writes must land for 7+ days before BG-108 closes. Daily cron check: SELECT MAX(created_at) FROM audit_log and alert if > 24h.

Total estimate when reopened: ~10-12h across 3-4 sessions.

Open decisions (locked when reopened, not now)

  1. Fail-open vs fail-closed on audit write failure — proposed default fail-open + monitoring alert if any failures in 24h. May want per-event override for role changes + compliance overrides.
  2. Auth event capture mechanism — Supabase Auth Hooks (Pro plan, needs confirmation) vs app-side login handler. Webhook is cleaner.
  3. Action verb convention — proposed dot-separated namespaced (deal.updated, auth.login.success).
  4. Retention — TBD. Suggested default: 2 years for security events, 90 days for routine business mutations.
  5. PII in payloads — TBD policy. Suggested: no full row dumps; only changed-fields list + actor + entity FKs.
  6. Sensitive read logging (Scope C) — explicitly out of scope v1. Table design (existing audit_log) does not foreclose adding it later.
