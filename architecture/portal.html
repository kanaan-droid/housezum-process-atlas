Portal Architecture

Last verified: 2026-04-28 Status: Living Update triggers: src/app/api/inbound/, src/app/api/feeds/, src/features/portal-engine/, src/features/portal-simulator/, src/features/inbound-signals/, portal-related migrations Owner:** Will (solo) — review on every commit touching update triggers

Scope

Country-agnostic portal subsystem. Two halves:

  • Outbound (publishing): listings → portal feeds (Bayut XML, PropertyFinder JSON, planned SeLoger / Leboncoin / Bien'ici / Idealista / Immobiliare / Homegate). Adapter pattern + canonical mapper so adding a new portal is one adapter, not a codebase ripple.
  • Inbound (signals): portal webhooks → inbound_signals table → lead_sessions (72h windows) → CRM lead generation. PF / Bayut / SeLoger / Homegate / Idealista / Immobiliare WhatsApp leads all funnel through one shared shape.

Doesn't cover: market-data feeds (e.g. Dubai Pulse / DLD — see grandfather note for BL-118 in scripts/docs-check.mjs). Different shape (outbound third-party data ingest vs portal-listing distribution); when 3+ market-data tickets land, propose market-intelligence.md.

Portal Simulator

BL-53 (D-93) — Portal Publishing Simulator. Canonical: docs/specs/BL-53-PORTAL-SIMULATOR-SPEC.md.

  • Full-screen "Preview on Portals" modal on listing detail. Two tabs: - Field Coverage — 🔴/🟡/🟢 issue list, readiness % + ranking quality % per portal. Each issue has a Fix → deep-link that returns to the simulator after edit. - Per-portal preview — listing card, title preview, photo order strip (Bayut + PropertyFinder today).
  • DB-driven validation rules: portal_field_requirements table holds the rules (38 UAE rules at ship, country-extensible). portal_community_mappings (40 Dubai entries) maps internal community names to portal-specific community IDs.
  • portalValidationService.ts is pure-TS — no DB writes during simulation. Reusable for future cron bulk-health checks.
  • New columns added to properties for portal needs: property_reference_id, rent_frequency, security_deposit, project_name, listing_metadata JSONB. listing_metadata is the extension-bag pattern (no migration for new portal-specific fields).
  • Architectural significance: validation rules are data (rows in portal_field_requirements), not code. Adding a market means inserting rules, not writing files. Same lookup-map principle as i18n.md and multi-tenancy.md § Country-agnostic.

Portal Publishing Engine

BL-55 (D-95) — full engine to network edge. Canonical spec: docs/specs/BL-55-PORTAL-PUBLISHING-ENGINE-SPEC-v2.md.

  • src/features/portal-engine/ is the home. Components: - canonicalMapper — DB row → CanonicalListing v2. Auto-detects market from city. The canonical model is the contract every adapter consumes. - communityNormalizer — internal community names → portal community IDs (via portal_community_mappings). - BayutAdapter — XML, DB-driven validation, ranking score, simulation persistence. Wires walkthrough share URL into &lt;VirtualTourURL&gt; (cross-link: photo-intelligence.md § Walkthrough subsystem, BL-89). - PropertyFinderAdapter — JSON, PF field codes. Wires walkthrough URL into virtual_tour_url. - Adapter registry — keyed by portal name; portalEngine orchestrator looks up the adapter. - feedSerializer — bulk XML merge + JSON serialization. - feedRunService — portal_feed_runs CRUD (run history, status tracking). - portalEngine orchestrator — simulate / runFeed / generateFeed entry points.
  • Feed API route: /api/feeds/[orgId]/[portal] serves XML/JSON with 1-hour cache. Bayut polls this URL (the agency owner configures Bayut to point here; Bayut pulls).
  • Admin Portals Dashboard at /admin/portals: portal status cards (Inactive / Mock / Live badges), Run Feed / Generate XML / Copy URL buttons, common-failure aggregation (top 5 blockers across all listings), feed health table per-listing, feed preview modal (dark XML preview, download).
  • Architectural significance: the adapter pattern is the country-agnostic primitive. Each portal adapter conforms to a single interface (PortalAdapter — validate(listing) → ValidationResult, serialize(listings) → string, rankingScore(listing) → number). New portals don't ripple: add an adapter, register it, no other code changes. The same interface contract is what makes BL-53 simulator work — it consumes adapters as a black box.
  • Dry-run discipline (BL-55 lesson): dryRun=true is safe for page load (no DB writes). "Run Feed (Mock)" writes real portal_feed_runs records — agents can audit what was sent. Pattern: any "preview vs ship" surface must default to dry-run; explicit user action triggers the writing path.
  • 5 new DB tables with RLS via get_user_org_id() (portal_field_requirements, portal_community_mappings, portal_feed_runs, plus simulation persistence tables — see DB-SCHEMA.md).

Signal Pipeline

BL-90 (D — see ROADMAP) — inbound signal pipeline spec. Canonical: docs/specs/BL-90-SIGNAL-PIPELINE-SPEC.md. Curl test guide: docs/testing/BL-90-CURL-TEST-GUIDE.md.

  • Inbound webhooks at /api/inbound/{property-finder, bayut, seloger, homegate, idealista, immobiliare, whatsapp}/route.ts. Each is a thin adapter that: 1. Validates the webhook secret (fail-closed via validateWebhookSecret.ts — see multi-tenancy.md § Security). 2. Validates the body via Zod (BL-92 hardening — Zod UUID on org_id, schema on payload shape). 3. Normalizes the portal-specific payload into the canonical signal shape. 4. Inserts into inbound_signals. 5. Routes into lead_sessions (72h windowing — same lead from same channel within 72h is one session, not two leads).
  • inbound_signals table — append-only ledger of every webhook event (parallel to activity_events in CRM, but for portal inbound). Source-of-truth before lead deduplication.
  • lead_sessions (72h windows) — deduplicates the same buyer hitting the same channel multiple times. Within a 72h window, additional webhook events extend the session; outside, a new lead.
  • shadow_listings — tracking listings that exist on portals but not (yet) in our DB. The Signals queue surfaces these so admins can resolve (link to existing listing or create new).
  • Status enum on inbound_signals.resolution_status: review_queue, unresolved, partial, resolved. The Signals queue (/admin/signals) filters by these. partial is valid (portal proxy contacts have no durable identity yet); resolved is correctly excluded from the queue.
  • Architectural significance: the funnel narrows. Webhook → inbound_signals (broad) → lead_sessions (deduplicated) → leads table (qualified). Each step has its own table, its own RLS, its own resolution state. Don't shortcut by writing webhooks directly to leads — the audit trail in inbound_signals is what lets agents debug "why didn't this lead show up?" weeks later.
  • Cross-link to STRATEGY.md § Signal Layer for the architecture-level framing of why this funnel exists at all (one of the four strategic layers).

Country-agnostic primitives

This is the cross-cutting theme that ties Simulator + Engine + Signals together.

  • One adapter per portal, conforming to a single interface. Country-extensibility means adding adapters, not adding country branches inside existing code.
  • Validation rules as data, not code (portal_field_requirements).
  • Community mappings as data (portal_community_mappings).
  • Canonical mapper auto-detects market from city — the adapter doesn't need to know what country it's in; the listing carries that signal.
  • Webhook handlers are thin — the canonical signal shape is the integration contract; portal-specific quirks live in the handler, not downstream.
