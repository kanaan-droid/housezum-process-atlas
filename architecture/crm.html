CRM Architecture

Last verified: 2026-04-28 Status: Living Update triggers: src/features/deals/, src/features/contacts/, src/features/leads/, src/features/viewings/, src/features/intel/ Owner:** Will (solo) — review on every commit touching update triggers

Scope

The CRM core: deals, contacts, leads, viewings, Intel Cards, lead qualification, deal-document linking. Distinct from:

  • multi-tenancy.md — RLS plumbing, org_id discipline, RBAC, security boundary patterns.
  • signing.md — document signing lifecycle (templates → recipients → PDF rendering → fully-executed). The deal-side of a signed document (Documents tab, signature status surfaces) lives here; the signing pipeline lives there.
  • agentic.md — L0/L1/L2 autonomy + Action Cards. Intel Cards (here) ≠ Action Cards (agentic.md) — see § "Intel Cards".

CRM owns the polymorphic party + activity ledger primitives that everything else builds on. If you're touching deal/contact/lead schema, entity_parties and activity_events are the load-bearing tables (BL-52). Read § "V2 schema foundation" first.

V2 schema foundation

The most architecturally significant ticket in the cluster: BL-52 (D-86) introduced the v2 polymorphic foundation that the rest of the CRM depends on.

  • entity_parties — polymorphic junction table with entity_type + entity_id + contact_id + role + is_primary + signature_required + signed_at + ownership_pct + notes + created_by. Replaces the older document_contacts and (alongside) deal_participants patterns. Invariant: any "who is involved with this thing" link goes through entity_parties. The polymorphic key is (entity_type, entity_id). Don't introduce parallel junction tables for new entity types — extend entity_parties with a new entity_type value.
  • activity_events — append-only ledger, indexed by entity, contact, deal, org, occurredat, eventtype. Invariant: append-only. Never UPDATE or DELETE here; corrections go in as new events. The Deal Timeline (BL-46) and Contact Activity tabs both read this.
  • viewing_feedback column additions (post-BL-52): viewing_id, contact_id, agent_user_id, notes, next_step. Real column names diverged from the original spec — buyer_interest_level, price_feedback, actual_visitors are the canonical names. Code that reads viewing feedback must use these, not spec names.
  • deal_participants polymorphism (relevant to BG-16 lesson, kept here as architectural context): rows in deal_participants carry either user_id (when the participant is an internal agent) or contact_id (when the participant is an external party — seller, buyer, buyeragent). Never both. Code that joins or formats dealparticipants must null-guard both sides — uid.slice(0, 8) on a null user_id was the canonical regression (BG-16).
  • All v2 tables use RLS via get_user_org_id(). Cross-link to multi-tenancy.md § RLS patterns.

Lead pipeline

BL-36 (D-83) and BL-59 (D-100) form the lead-side ingestion → conversion pipeline.

  • BL-36 — Lead Qualification Form. Public token-based /q/[token] form (no auth). Typeform-style 4-screen flow: budget bands → buying purpose → timeline → must-haves (140 char). GDPR consent checkbox for EU orgs (FR/CH/IT) — the country-aware consent gate is canonical for any future public form. Invariants: - Token expiry 30 days (matches the enrichment-form pattern — keep these consistent so agents can train on one mental model). - On submission: lead.status = 'qualified' is system-only. Agents cannot manually set 'qualified' — it's reserved for form-submission events. This prevents agents from gaming intent score and ensures every "qualified" lead has the corroborating answers. - intent_score boost formula: +15 budget, +20 ready-now timeline, +10 must-haves, +5 purpose. Documented here so future scoring tweaks have a baseline. - lead_qualified Intel Card goes to the assigned agent, not the admin who sent the qualification form. Routing to the sender breaks the pattern when admins send on behalf of agents.
  • BL-59 — Lead → Contact conversion. New requirements table, converted lead status, ConvertLeadModal (3-step), Requirements tab on contact detail, /api/leads/convert endpoint. Lesson from BL-59b spec work: buyer_profiles and seller_profiles are real DB tables, not UI-only. Always check DB before retiring UI surfaces. buyer_profiles and requirements overlap significantly — requirements is the consolidated home for buyer demand profiles. seller_profiles stays (asset-side concern; the real seller fix is TD-12 properties → assets+listings split).

Reverse Match

BL-62 (D-101) — buyer requirements → property matching on Property Detail.

  • Two-query approach: requirements (active buyers) + contacts (for display). No joins — see WAYS-OF-WORKING § "PostgREST + RLS" for why FK joins across RLS-protected tables fail (returns empty silently).
  • Client-side scoring across 4 dimensions: budget 40%, bedrooms 25%, area 25%, property type 10%. Org-wide matching. Top 10 results, threshold ≥40. Strong (≥70) / Partial (40–69) tiers.
  • Why client-side scoring: deterministic, easy to tune from logs, no LLM cost, no database function maintenance. The scoring weights are config — change them in code, no DB migration.
  • Pre-filled WhatsApp + email outreach. Spec: docs/specs/BL-62-REVERSE-MATCH-SPEC.md. STRATEGY.md mentions Buyer-Seller Matching as part of the Brain layer — Reverse Match is the agent-facing surface; the autonomous matching agent (when it lands) will use the same scoring primitives.

Deal Health Score

BL-71 — dealHealthService.ts (Jest-tested per BL-76, see WAYS-OF-WORKING § Testing Protocol).

  • Composes from multiple signals: stage age, doc completeness, last-activity recency, missing parties, viewing-feedback sentiment.
  • Architectural significance: dealHealth is a pure-compute service (no DB writes). Inputs are deal + activityevents + entityparties; output is a single score. Lives in src/features/deals/services/. Pattern: any "scoring" feature in the CRM should be a pure-compute service — easy to test, easy to A/B by swapping config, no migration churn.
  • Decay: exponential decay on signal age (see scoreDecayService Jest tests for the formula). Decay is the canonical answer for "old signals are less informative".

Intel Cards

BL-69 — Intel Cards UX (whole-card-tappable for primary CTA + stage_suggestions card type).

  • Intel Cards ≠ Action Cards (this is the frequent source-of-confusion). Intel Cards surface CRM signals an agent should be aware of — "Lead X just qualified", "Property Y has 3 matching buyers", "Viewing Z had high interest, follow up". They live in src/features/intel/. Action Cards (covered in agentic.md) are agentic proposals the agent approves/rejects — "Send WhatsApp follow-up to X?", "Advance deal to Y stage?".
  • Whole-card-tappable pattern: primary CTA fires on card tap, secondary CTAs are explicit buttons. UX rule from BL-69 — clicks on the card body should always fire the most likely next action, not just navigate.
  • stage_suggestions card type (migration 20260314_bl69_stage_suggestion_card_type.sql): the Deal Orchestrator (BL-38) produces these to suggest stage advancement. Cross-link to agentic.md — Deal Orchestrator also feeds Action Cards in BL-97.

Deal creation

NU-21 — New Deal modal: seller auto-fill, buyer agent, deal title generation.

  • When the agent creates a deal from a property + contact pair, NU-21 wires: - Seller auto-fill from properties.seller_contact_id (BL-93 link). - Buyer agent detection from deal_participants if a buyer_agent row exists for the contact's prior deals — surfaces as a default but agent-editable. - Deal title generation — formulaic, e.g. "{property_short} – {buyer_name}". The format is config (in dealService.ts); change it once for global consistency.
  • Architectural significance: "infer from existing CRM relationships" is the core New-Deal UX pattern. New fields added to deals should follow the same shape — auto-fill from existing data when possible, present the agent with the populated value, allow override. Don't ship blank-form-then-validate.

SQL-only data correction. 6 EU properties had energy_efficiency_index landing exactly on class boundaries; rewrote index to safe-interior values, preserved stored class. Pre-fix diagnostic + post-fix verification: 0 mismatches across 15 in-scope EU properties.
