Signing Architecture

Scope

Architectural primitives, invariants, and cross-cutting patterns for the signing subsystem. Covers template definition, field placement, recipient routing, CRM auto-fill, PDF rendering pipeline, signature accumulation, and the fully-executed lifecycle. Canonical E2E flow is docs/specs/NU-7-SIGNING-SPEC.md — this doc complements that spec by capturing the why behind the architectural shape, not the user-story flow. Deal-side document linking (deal-document, contact-document) lives in crm.md.

Template canvas + field placement

Tickets here: BL-81, BG-21, BG-22, BG-23.

  • Multi-page PDF rendering is mandatory for both authoring (TemplateCanvas) and prepare (FieldPlacementCanvas). BG-22 rewrote TemplateCanvas from single-page to all-pages-stacked-vertically with per-page coordinate space and a page selector for field placement; BL-81 added the same multi-page navigation to FieldPlacementCanvas. Invariant: every canvas surface that sits over a PDF must support multi-page from the start — single-page-only is a future-debt trap.
  • PDF.js loader + CSP boundary (BG-21). PDF.js loads from cdnjs.cloudflare.com, which means the CSP must allowlist that origin in script-src, connect-src, and worker-src. When CSP changes (e.g. BL-92 hardening tightened defaults), the PDF.js allowlist must be re-verified — a missing entry breaks template rendering silently in production with no console error visible to the agent.
  • DocumentPrepareModal step structure (BG-23). Template card click must setStep(2) before awaiting any async fetch, otherwise the agent sees a frozen-on-step-1 UI while the network resolves. Pattern: synchronous step advance, then async data load with its own loading state. Wrapping the entire selectTemplate in try/catch is mandatory — fetch failures silently leaving the UI stuck on step 1 was the original bug.
  • x_pts/y_pts are NULL unless prepared via DocumentPrepareModal. Field positions in document_signature_fields only get populated when the agent runs through prepare. A signing request that skips prepare (or hits the legacy direct-send path before NU-20 was wired correctly) leaves coords NULL — code that reads coords must handle the NULL case explicitly, never assume positions exist.

Recipient routing + step structure

Tickets here: BL-81i, NU-17.

  • Recipient selection moved to step 2 (NU-17). Send buttons are conditional on recipient state — the agent cannot send until at least one recipient has the contact info needed for the chosen channel (email or WhatsApp). Conditional UI prevents the dead-end where send-fails-with-no-feedback because of a missing email address.
  • Recipient CRM pre-fill (BL-81i). When a recipient slot is assigned to a deal participant, the agent's name/phone/email pre-fill from deal_participants rather than typing. Excludes agent_prepare fields from the progress counter (BL-81i bug fix) — those are the agent's own fields, not recipient-input fields, and counting them broke the "all recipient fields filled?" gate.
  • agent_prepare progress counter must filter to recipient-input fields only. Agent's pre-prepared values aren't progress — they're the starting point.

CRM auto-fill

Tickets here: BL-81i, BL-81j, NU-16, NU-24.

  • Server-side via /api/signing/crm-autofill (BL-81j). The autofill route uses the service_role Supabase client to bypass RLS. Canonical reason: agent A is signing a document about a property listed by agent B; RLS on properties would block agent A from reading the property's contact data via the anon client. The route validates caller auth + scope, then bypasses RLS only for the contact lookup it needs. This is the canonical pattern for cross-agent CRM access — see multi-tenancy.md § Security for the broader service-role discipline.
  • Property lookup goes through deal_assets (NU-16), not via the property foreign key directly. deal_assets is the polymorphic join (BL-70 doc architecture) — properties, units, listings can all be assets. Looking up via property FK misses non-property asset types and breaks for upcoming asset variants.
  • Role-aware contact_id, case-normalised (NU-24). When DirectSendSheet maps a field to a contact, it: (1) determines role from the field's role_label, (2) normalises both sides via .toLowerCase() to bridge the Title Case (role_label) ↔ lowercase (deal_participants.role) gap, and (3) aliases "agent" ↔ "listing_agent" in both directions. The first two are required because the case mismatch is silent — neither side errors, the lookup just returns null. The alias is required because templates were authored against "Agent" but DB stores "listing_agent".
  • acroform_values: {} on secondary sends (NU-24). When the same document gets resent (e.g., correction path), the autofill payload must clear acroform_values to {} rather than carry over the prior send's values. Invariant: never overwrite shared document values on a resend; the agent re-prepares fresh, the prior sign's values stay attached to that request's document_signature_fields rows.

Cross-link: multi-tenancy.md § Service role vs. authenticated client for the BG-2 split-clients pattern that underpins this route.

PDF rendering: draw-then-strip

Tickets here: NU-8, NU-10, NU-11, NU-27, BG-21, BG-35. The most architecturally interesting section.

The recipient-side /sign/[token] page renders a PDF that mixes (a) the original document's page content, (b) AcroForm widgets at field positions for inputs, (c) static rendered values from prior signers and from the agent's prepare phase. The naive approach — keep widgets and form.flatten() at the end — silently fails on PDFs produced by reportlab and pdfrw (a substantial fraction of real-world templates). The architectural answer is draw-then-strip.

Draw-then-strip pattern

  1. Load the base PDF: PDFDocument.load(bytes, { ignoreEncryption: true }). (Encryption flag is mandatory — many real PDFs ship with an empty owner password that breaks default-load.)
  2. For each field whose value should be visually baked into the page (agent-prepared values; values from prior signers; not-recipient-owned fields): - Use page.drawText(value, { x, y, size, font }) directly onto the page content stream. - Do NOT manipulate the widget annotation.
  3. After all values are drawn: - Strip every Widget annotation from every page's /Annots array. - Remove the /AcroForm entry from the document catalog.
  4. Save: serialise via await pdfDoc.save(). Critical: extract the underlying ArrayBuffer correctly with saved.buffer.slice(saved.byteOffset, saved.byteOffset + saved.byteLength) — the raw .buffer reference can include trailing bytes from a shared Uint8Array allocator and corrupts the response.

Why widgets are stripped after the first signer

Once the document is signed by signer 1, the AcroForm widget annotations are stripped (step 3 above). For signer 2, there are no widgets to draw text through — widget-based text drawing is impossible. Therefore: rebuild from the base PDF on every signer submission. Always fetch ALL field values + signatures across ALL document_signature_requests for the document, then rebuild in one pass from the original base. Stacking signed PDFs (signer 2 on top of signer 1's signed PDF) is the wrong shape — it works for two signers and breaks for three. The rebuild-from-base discipline is invariant.

Recipient-owned fields are excluded from baking

Per-recipient field filtering is enforced at PDF-build time too: when fetching document_field_values, build an exclusion set of recipient-assigned field IDs and skip them during draw. The recipient sees their own widgets active; everyone else's prior values are baked into the content stream as drawn text. Agent-prepare overlays (displayValue rendered in the editor) return null on the recipient view (BG-35).

NU-8: PDF proxy route

/api/signing/[token]/pdf does server-side bucket detection (base-documents vs documents) and applies acroform_values server-side via pdf-lib before streaming the filled PDF to the browser. Eliminates client-side bucket detection entirely. The recipient browser never needs to know which Supabase bucket the base PDF lives in. Cache-Control is no-store during the active signing flow — max-age=300 was tested and produced stale PDFs during debugging. Production keeps no-store even outside debug.

NU-10 + NU-11 (D-147): the canonical draw-then-strip implementation

NU-10 originally framed the problem as "fix form.flatten()"; the architectural answer was that flatten was wrong for our shape and the proxy route should draw text directly, strip widgets, and remove /AcroForm from the catalog. Recipient-owned fields excluded from baking (NU-11). agent_prepare overlays return null on recipient view. Text field overlays use flex centering, boxSizing, correct padding. Double-text (the bug the original NU-10 report named) was eliminated as a symptom of the rewrite, not as the headline. The D-147 ROADMAP entry is the canonical reference for this implementation.

NU-27: signer text fields render into the final PDF

Passport, nationality, and similar text fields entered by the signer on /sign/[token] must be rendered into the final PDF — not just persisted in document_field_values. Implementation: on submit, fetch the just-saved values and pass them through the same drawText-then-strip pipeline as agent-prepare values. Without this, the signed PDF is missing all signer-text content and the document is non-compliant downstream.

BG-35: PDFFormEditor overlay rules

Overlay (the displayed value rendered over the input widget) computation rules: compute displayValue (not raw value, which may not match what the user sees), use visible overflow (overflow: visible on overlay container — overflow: hidden clips the value text in narrow boxes), let the rendered value replace the label rather than stack alongside. Email-first label matching (when a field's purpose is ambiguous, prefer email pattern over name pattern). Prefill order matters — "Email Address" contains "name" as a substring; the wrong match-order returns the wrong field.

Other constants from session memory worth preserving

  • BASELINE_NUDGE = 0.08 (8% of fontSize) — empirically derived offset for drawText to align with widget baseline; hardcoded +2px produces visible drift.
  • Use instanceof PDFTextField / instanceof PDFCheckBox for type checks. Never constructor.name — class names are mangled in production builds, the check returns false, the field falls through, the bake silently skips.

Signature accumulation

Tickets here: NU-25, NU-19.

  • NU-25 invariant: never form.flatten(). Each submit builds on the previous signed PDF. Always use the existing signed_file_path as source PDF if it exists; otherwise the base PDF. Stack signatures by drawing each new signature onto the current source-of-truth PDF, then strip widgets again, then save. The signature accumulator is the single canonical place where signatures are composed; it must be the only path from "submit" to "stored PDF" so the rebuild-from-base discipline holds.
  • Signed PDF retrieval via signed URL when is_signed=true (NU-19). When the agent or recipient downloads a "fully executed" document, the route serves the signed PDF via a Supabase signed URL with limited expiry — not the raw bucket path. Avoids both leaking bucket structure and serving expired-cookie URLs.

Fully-executed lifecycle

Tickets here: NU-18, BL-102.

  • Agent auto-sign at send time (NU-18). When the agent sends a document, their own signature is captured and applied immediately — the agent doesn't need to also receive a signing request and click through. Fully executed requires (a) all recipient requests signed, AND (b) agent signature present. Either gate alone is insufficient.
  • document.is_signed = true only when ALL non-cancelled requests are signed (BL-81c). The is_signed boolean is computed, not directly written — it's a function of the request rollup with cancelled requests excluded. Cancelled requests don't block "fully executed".
  • BL-102 lifecycle ops: void / resend / correct. All three operations live under /api/signing/[token]/{void,resend,correct} and snapshot the document state into document_signing_versions before mutating. The snapshot captures field values BEFORE the upsert, not after — important when reconstructing history. - Resend creates a new request with parent_request_id pointing at the original; the original gets cancelled. - Correct is conditional on a is_material toggle on signing_template_fields — only fields marked material trigger a correction snapshot + auto-resend; immaterial edits update in place. The toggle is set at template-canvas authoring time. - Diff highlight on the sign page surfaces what changed between the corrected request and its parent; the version drawer in the agent surface lists every snapshot.
  • Expiry reminder cron (BL-102) lives under vercel.json cron jobs; flagged as an aws-migration item per STRATEGY.md § "Non-negotiable architectural rules" (every Vercel cron must have a Lambda equivalent on AWS migration).
  • Overflow menu must dismiss before async resend call (BL-102 ROADMAP entry). Click-through pattern: close menu synchronously, then await async — otherwise the menu state stays open during the network round-trip and another tap re-fires the action.

Direct-send pattern

Tickets here: NU-13, NU-20, NU-24.

  • DirectSendSheet skips all modal steps (NU-20). Entry from the deal Documents tab → fast path that takes a pre-prepared template and a contact, skips the four-step DocumentPrepareModal flow, and creates the signing request directly. Bundled DancingScript font ships with the app (in public/fonts/DancingScript-Regular.ttf) for typed signatures — the font must be local because external CDN fonts violate the offline-capable signing requirement and the CSP discipline.
  • Field assignment edge cases (NU-24). DirectSendSheet's field-to-contact mapping uses the same role-aware, case-normalised, alias-bidirectional lookup as the modal path — see § "CRM auto-fill" above. The acroform_values: {} clear-on-secondary-send invariant lives in DirectSendSheet's send handler too.
  • Copy signing link on pending requests + awaiting count (NU-13). Agents copy the public /sign/[token] URL from the deal Documents tab and paste it in a different channel (e.g. WhatsApp) when email delivery fails. The "awaiting" count fix in NU-13: the count must reflect non-cancelled, non-signed requests only — earlier display included cancelled requests in the awaiting count, surprising the agent.

Edge cases & invariants

Pulled forward as a single bullet list so the reader can ctrl-F. Most have a canonical home in WAYS-OF-WORKING.md § Signing Architecture Learnings; cross-link there for any rule whose application reaches beyond signing. The "why" lives here.

  • credentials: 'include' on every signing fetch (BG-24). Auth cookies aren't sent on same-origin API requests on some Vercel deployment configurations without explicit credentials: 'include'. BG-24 was the canonical instance: signing send hung indefinitely because the API route returned 401 silently. The fix propagated to all signing fetch sites and was generalised in WAYS-OF-WORKING.md § Fetch Credentials Rule (Non-Negotiable).
  • role_label is Title Case ("Seller" / "Buyer" / "Agent"); deal_participants.role is lowercase ("seller" / "buyer" / "listing_agent"). Always .toLowerCase() when mapping. Always alias "agent" ↔ "listing_agent" bidirectionally — templates were authored Title Case + short-form, DB stores lowercase + listing_agent.
  • AcroForm widgets are stripped after first signer signs. Rebuild from base PDF for every submission; never stack signed PDFs.
  • acroform_values: {} on secondary sends — never overwrite shared document values across resends.
  • x_pts/y_pts are NULL unless prepared via DocumentPrepareModal. Code reading positions must handle NULL.
  • Always use existing signed_file_path as source PDF when it exists — accumulate signatures from there.
  • AcroForm hydration: never hydrate the focused field. Use document.activeElement check (hydrateFromPropsSkipFocused); use change listener (not input) so values capture only on blur.
  • createPortal for full-screen modals. Modals nested in fixed-positioned parents (e.g. deal drawer) trap z-index unless rendered via createPortal(jsx, document.body).
  • Per-recipient field filtering by assigned_contact_id at API-route-creation time. Each recipient's request gets only their own fields plus any agent fields.
  • Stale closures in submit handlers. useCallback deps must include all state used inside — fieldValues, signedFields, etc. — otherwise stale closures silently submit empty data.
