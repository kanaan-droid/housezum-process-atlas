Q&A response cache (BL-106)

Why it exists

The Q&A engine's cold path runs intent matcher → context assembler → data sieve → answer polisher, all of which involve LLM calls. Cold avg latency was ~3.3s. The cache is a pure memoization layer over the engine's top-level entry point — same input, same output, returned in <50ms instead of seconds.

Measured impact at ship:

  • Warm hit rate: 86%
  • p50 latency: 4,277ms → 761ms
  • avg latency: 3,783ms → 1,342ms

Cache key

Algorithm: SHA-256 hex over a canonical JSON of the input tuple.

Key shape: qa:v1:{sha256-hex}

Inputs hashed: | Field | Source | Empty value | |-------|--------|-------------| | q | normalized question | — | | p | property_id | '' | | d | deal_id | '' | | c | org_country_default | — |

The v1: prefix is a deliberate version namespace. If we change normalization, key inputs, or response shape, bump to v2: and old keys age out via TTL — no flush required.

Why light normalization (not stemming, not filler-word stripping)

Question normalization is deliberately minimal: lowercase + trim + collapse internal whitespace. Nothing else.

Aggressive normalization (stemming, removing "the"/"a"/"is", synonym substitution) risks collapsing semantically-different questions onto the same cache key. "What is the price?" and "What was the price?" are different questions with different correct answers in the deal-history context.

The cache is a memoization layer, not a semantic similarity layer. Synonym handling is the intent matcher's job upstream. The cache only catches identical (or near-identical) re-asks.

TTL strategy

ScopeTTLRationale
Scoped (deal_id or property_id present)15 minDeal/property data mutates; bounded staleness window keeps cached answers from drifting too far from reality
Org-only (no deal/property)24hOrg-level KB and country defaults are effectively static within a day

TTL is selected by pickTtl() based on whether either scope ID is present in the input.

What's NOT cached (guardrails)

The isCacheable() predicate gates all cache writes. Two response shapes are skipped:

  • answer_type === 'clarification_needed' — these are low-confidence misses where the engine is asking the user for more info. Caching them locks in the bad classification: every future asker of the same question gets the same "I need more context" response, even after the underlying data improves.
  • data_gap_flag === 'stale' — staleness is a signal that the data has moved past its freshness window. Caching a stale answer defeats the entire freshness model.

Everything else is cacheable, including fallback/null and fallback/nyt (not-yet-tracked) responses. Those are legitimate, deterministic outputs of the engine's data sieve.

The schema fallback path (BL-115) also writes nothing to the cache by design — it's an escape hatch for column-level reads where caching the wrong shape early would be hard to invalidate.

Pipeline placement

The cache check sits at the very top of the engine entry point (handleQaRequest in qa-engine.service.ts) — before the intent matcher, before context assembly, before any LLM call. On hit, the response returns immediately with match_source overridden to 'cache' so downstream observability can distinguish cache hits from cold runs.

The cache write sits at the very bottom — after the answer polisher has produced the final user-facing string. We cache the polished response, not intermediate state.

Failure mode

Fail open. Both getCachedResponse() and setCachedResponse() wrap their Redis calls in try/catch, log a warning on failure, and return null/void. Redis being slow, unreachable, or rate-limited must never break a Q&A request — the engine just runs cold.

This is enforced at the primitive level: nothing in the call site needs to know about cache failures.

Invalidation

v1 has no active invalidation. TTL is the only safety net. This is a deliberate v1 simplicity choice.

Operational consequence: if a deal is updated (price change, new participant, status flip), cached answers reflecting the old state remain valid for up to 15 minutes. For most Q&A surfaces (viewing mode, property page assistant) this is acceptable — agents are not in the loop at sub-15-minute latency.

When to add active invalidation, plus the recommended Supabase Database Webhook → /api/internal/cache-invalidate → redis.del(pattern) flow, is documented in WAYS-OF-WORKING.md § "Cache Invalidation Mental Model (BL-106)".

Extending the cache to new features

When adding caching to a new service (BL-97 action card resolution, market analysis, anything slow + idempotent):

  1. Don't reuse qa-cache.service.ts — it's tightly typed to QaResponse. Create a sibling service (action-card-cache.service.ts, etc.).
  2. Reuse the patterns: - Redis.fromEnv() client — same Upstash instance, no new provisioning needed - SHA-256 key with versioned prefix ({feature}:v1:) - Fail-open get/set (try/catch → log + return null/void) - isCacheable(resp) guard for low-confidence or stale results - TTL picked by scope, not hardcoded at the call site
  3. Pick TTLs deliberately. The cache is a correctness/freshness tradeoff. Default to short.
  4. Add a section to this doc mirroring the BL-106 structure (Why / Key / Normalization / TTL / Guardrails / Placement / Failure / Invalidation).
  5. Update WAYS-OF-WORKING.md with new operational notes — flush command if the key prefix differs, invalidation trigger if applicable.

What to watch

  • Free tier ceiling. 500k commands/month and 256MB. Current Q&A volume is well within budget, but adding cache use cases multiplies command volume. Upgrade trigger: monthly commands cross ~80% of the limit, OR eviction starts firing visibly (cold rate climbing without TTL expiry).
  • Active invalidation pressure. The 15-min scoped TTL is fine today. The first user-visible "I edited the deal but the assistant still says the old price" complaint is the signal to ship the webhook-based invalidation pattern from WAYS-OF-WORKING.
  • Cache stampede. No single-flight protection today. If a popular question's TTL expires under load, every concurrent request runs the engine cold. Acceptable at v1 traffic; revisit at scale.
  • Single-region latency. Frankfurt-only. UAE users hit transatlantic latency on every Redis op (~80–100ms). Still beats cold engine runs, so acceptable today. Revisit if/when we provision a UAE-regional Upstash instance, or if the multi-region story for the broader app firms up.
  • Locale-keyed responses. Cache key currently includes org_country_default but NOT user locale. With BL-120 i18n shipped and BL-121 (AR) in flight, this becomes wrong as soon as Q&A responses start being locale-aware (BG-72). Add locale to the key inputs and bump to qa:v2: when BG-72 lands.
