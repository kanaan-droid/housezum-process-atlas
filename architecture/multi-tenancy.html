  • BL-22 — Multi-org support (org switcher, multiple memberships per user, switch context without logout) — [no commit, ROADMAP-only]

Long-running architectural theme. A user (auth.users.id) can be a member of multiple orgs via org_members rows; each membership carries its own role (owner/admin/manager/agent/assistant/viewer) and permissions_override. The org switcher UI (when built) reads the active org_id from session/cookie — DB queries are always scoped by the active org via RLS. Invariant: never grant a user implicit access to all their orgs at once; the active org is the only org RLS evaluates against per request. Cross-org data fetches require an explicit elevation pattern (service-role with org_id parameter) and are the exception, not the rule (see § "Cross-org operations").

Absorbed tickets

  • BG-2 — Invite status update fix (split-clients pattern) — files: src/app/api/invite-signup/route.ts, supabase/migrations/003_drop_legacy_users.sql

Canonical pattern for service-role hygiene. signInWithPassword() was tainting the service role client's auth state, which then caused RLS to block the invitation UPDATE. Fix: split into two Supabase clients — supabaseAdmin (clean, for DB ops) + supabaseAuth (for sign-in only). All sb references replaced. Invariant: never share a single Supabase client between auth-state operations and service-role DB ops in the same request. The auth state on a service-role client is incidental but RLS evaluates against it; mixing the two breaks RLS bypass silently. Every API route that calls both auth.signInWithPassword/auth.exchangeCodeForSession AND a privileged DB op must use two separate clients.

  • BL-64 — Service-role PATCH bypass on agent_profiles (canonical pattern) — files: src/app/admin/members/[userId]/page.tsx, src/app/api/admin/members/[userId]/route.ts

When an admin needs to update another agent's agent_profiles row, the client-side update fails RLS (admins are in the same org but RLS policy on agent_profiles restricts updates to self). Fix pattern: route the update through a server-side API route that uses the service-role client, with explicit role + org checks before the UPDATE. The route validates: (1) caller is authenticated, (2) caller's orgid matches target's orgid, (3) caller has admin/owner/manager role. Then service-role bypass executes the UPDATE. This is the canonical "admin operates on another user's data" pattern — replicate for any similar future admin surface (e.g., updating another agent's preferences, role, or status).

Security

RLS, Zod input validation, service-role boundaries, and data-safety guardrails together form the boundary-enforcement layer of multi-tenancy. RLS prevents cross-org reads/writes; Zod prevents malformed input from reaching the DB; service-role boundaries prevent unintentional privilege escalation; data-safety guardrails prevent destructive operations from cascading.

Tier 1 hardening (BL-92, D-166)

  • validateWebhookSecret.ts — fail-closed webhook secret validator. Applied to 6 portal routes (PropertyFinder, Bayut, Homegate, SeLoger, Idealista, Immobiliare under src/app/api/inbound/). Rule: webhooks fail closed, never fail open. If the validator can't verify the secret (missing env var, mismatch, etc.) the route returns 401 without inspecting the payload. Webhook validation precedes Zod validation precedes business logic.
  • Zod UUID on org_id for the same 6 portal routes. Catches malformed payloads at the boundary; never trust webhook input shape.
  • Zod body on qualification POST — public /q/[token] form submission validates the payload before writing to lead_qualification_requests.
  • CSP + security headers in next.config.js. Header-based, not meta-tag-based (per the BL-92 hardening session: meta-tag CSP is unreliable across CDN edge cases). Allowlist for Google Fonts + randomuser.me (BL-92b adjustment); cdnjs.cloudflare.com for PDF.js (BG-21 fix).
  • Spec: docs/specs/BL-92-SECURITY-SPEC.md. Tiers 2–4 (authenticated routes deeper review, GDPR, AI-layer hardening) deferred at BL-92 close; partially addressed by BL-92b.

Data Safety Guardrails (BL-103, D-177)

  • photo_annotations foreign key changed to ON DELETE RESTRICT (was CASCADE). Protects annotation training data from accidental cascade-delete when a parent property is removed. Rule: any FK pointing at training/labeled data uses RESTRICT, not CASCADE. The data has higher protection than the parent row.
  • --dry-run flag added to seed-dubai.mjs and seed-corpus.mjs. All destructive seed operations are guarded — running without --dry-run=false (or equivalent confirmation flag) is a no-op. Rule: any new destructive script — wipes, mass updates, cascade deletes — implements --dry-run as the default. Real execution requires an explicit override flag.
