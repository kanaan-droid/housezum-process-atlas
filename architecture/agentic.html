  • BG-45 — Action Log expanded view missing execution details — files: src/app/action-log/page.tsx, src/app/admin/action-log/page.tsx

L2 auto-executions persist execution_result in ai_action_cards with full payload (Twilio SID, recipient phone, channel, error class) but the expanded card view only renders the action's user-visible summary. Surfacing all execution_result keys in the detail section is necessary for agents to debug their own L2 deliveries (e.g., "did this WhatsApp actually go out, and to which phone?") without needing admin help. Pattern: render execution_result as a flat key/value list under the existing summary, not as collapsed JSON.

Absorbed tickets

  • BG-36 — Sync auth email to agent_profiles (backfill, trigger, seed script fix, shared helper) — files: scripts/lib/getAgentByEmail.mjs, supabase/migrations/20260411_bg36_agent_profile_email_sync.sql

Foundational invariant for action-card execution: the agent identity used to dispatch a card must be reliably resolvable from auth.users via email. agent_profiles.email is not authoritative on its own — it can drift from auth.users.email between profile edits. The migration adds (a) a backfill from auth.users and (b) a trigger that keeps agent_profiles.email in sync. Seed scripts use the shared getAgentByEmail() helper rather than ad-hoc lookups. Anywhere downstream code needs an agent identity for action-card dispatch, route through this helper — direct queries on agent_profiles.email are not safe.

  • BG-37 — Action card routes 401 fix: use supabase.auth.getSession() for token instead of localStorage — files: src/features/agentic/components/ActionInbox.tsx

Browser clients in this project hold the Supabase session via supabase.auth.getSession(), not in localStorage. Earlier code attempting to read the JWT from localStorage returned null, so the action-card API routes (which expect a Bearer token) returned 401. Two corollaries documented in multi-tenancy.md § Security: (1) API routes must use the service-role createClient for auth.getUser(token) server-side, never the anon browser client; (2) any client-side code that needs the JWT goes through supabase.auth.getSession(). Cross-link: this pattern applies to every authenticated API route, not just agentic.

  • BL-111 — Seed script: agents must not appear as contacts (dual-identity quirk) — files: scripts/seed-dubai.mjs — [no commit, ROADMAP-only]

Discovered during BL-97 Phase D L2 auto-execute testing. seed-dubai.mjs was creating contacts rows for agent personas, so the same person appeared both as an org_members row (agent who logs in) AND as a contacts row (CRM contact who can be a recipient). When an action card targeted that contact_id, the executor would surface confusing dual-identity behavior (the action would route to the agent's own WhatsApp). Fix: before inserting into contacts, check if user_id exists in org_members for the same org and skip if so. This is the canonical "agent identity ≠ contact identity" rule for the seed/test data layer, and a reminder that L2 auto-execute is not safe against pathological seed data.
