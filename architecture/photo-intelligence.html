Annotation pipeline

BG-25 — annotation queue cascade fixes. The headline bug ("queue shows 0/0") had four overlapping causes that surfaced as a single symptom; the fix is canonical for several React/state patterns.

  • Counter query target. photo_evaluations (empty in dev) ≠ photo_annotations corpus. The counter must query the same table the loader reads — naming-similar but distinct tables silently agree on emptiness.
  • credentials: 'include' on the queue API call. Same systemic pattern as BG-24 in signing — Vercel same-origin auth cookies don't propagate without explicit credentials. See WAYS-OF-WORKING § "Fetch Credentials Rule".
  • useCallback stale closure on orgId. loadQueue was bound to a stale orgId because the dep array didn't include it. Pattern: useCallback/useEffect deps must include every value used inside; stale closures silently submit/load against undefined or prior values.
  • loadQueue fired before orgId resolved. OrgProvider resolves async; effects that depend on orgId must guard if (!orgId) return or wait for orgLoading=false.

Architectural significance: these four are general patterns, not annotation-specific. The same shape recurs in any "load-on-mount, requires auth + org context" component. BG-25's checklist is the canonical diagnostic when an admin loader silently shows zero.

Room detection

BL-86 — auto room detection on photo upload via Claude Vision (Haiku model).

  • Fire-and-forget: room classification fires after upload completes, doesn't block the agent. ~$0.0003/photo at Haiku pricing.
  • API route: src/app/api/photos/detect-room/route.ts. Component integration: PhotoGallery.tsx.
  • Result writes a room_type tag onto the photo row.
  • Architectural significance: the classifier is a swappable boundary. Today it calls Claude Vision via the central ai_service wrapper (per STRATEGY.md "AI wrapper only" rule). Tomorrow's CLIP/ViT migration (also per STRATEGY.md, gated on 5k labeled examples from BL-54b) swaps the implementation behind the same route boundary — the rest of the app cares only that detect-room returns a room tag. Don't leak Claude/Anthropic-specific shape into callers.

Walkthrough subsystem

BL-87 (D-157) — Media Intelligence: Walkthrough Reorder & Cover Photo. BL-89 (D-160) — Walkthrough Public Share URL + Portal Distribution.

  • walkthrough_orderings table persists the agent's saved slide sequence (migration 20260331_bl87_walkthrough_orderings.sql). The default order is AI-derived; agents can drag-reorder, and the saved order is delta-tracked against the AI baseline (training-signal capture).
  • buildSlidesOrdered (in walkthroughService.ts) bypasses the AI re-sort when a saved order exists. Invariant: when a saved ordering exists, never re-run the AI sorter — that erases the agent's choice and silently overrides their training signal. The walkthrough has its own ordering invariant separate from photo gallery ordering (which doesn't persist).
  • Cover photo selection (BL-87b): star button on grid cards + lightbox. The star is the only entry point — don't add cover-toggle UI elsewhere; this is the canonical surface.
  • Media event tracking: viewed, play_started, progress (25/50/75/100%), completed, dwelled. Persisted in media_events table. Training-signal capture for engagement-based reordering downstream.
  • BL-89 — Public share URL. Public /w/[token] page (no auth, full player, "Powered by HouseZoom" branding). Share button in WalkthroughPlayer (native share sheet / clipboard fallback). Token persisted in listing_metadata JSONB — no migration needed (the metadata-JSONB-as-extension-bag pattern is canonical for token-bearing fields that don't warrant their own column). Wired into Bayut &lt;VirtualTourURL&gt; and PropertyFinder virtual_tour_url via canonicalMapper — see portal.md for the adapter pattern.

Corpus catalog

BL-84 — 731 curated photo URLs merged into photo-data.mjs (bedroom +80, amenity +110, view +81, dining +81). BL-94 (D-165) — corpus property_id=null discipline.

  • Corpus photos vs property photos are intentionally separate. Corpus photos exist for ML labeling/training and have property_id = NULL. Property photos belong to a real listing and have a non-null property_id. The two share the photos table but the discipline is enforced at the seeder level.
  • BL-94 lesson (canonical): an earlier version of seed-corpus.mjs called getOrCreateCorpusProperty() which fabricated fake property rows to satisfy the foreign key. Those fake rows then leaked onto the Listings page. Fix and invariant: property_id: null is the only correct value for corpus inserts. Never fabricate property rows for the corpus side. The --wipe-corpus flag added in BL-94 is the safe-by-default discipline (per multi-tenancy.md § Security Data Safety Guardrails — destructive scripts require explicit override flags).
  • photo_annotations FK is ON DELETE RESTRICT (BL-103). Annotation training data outlives any single property. RESTRICT prevents cascade-delete loss when a property is removed. Cross-link to multi-tenancy.md § Security.
  • Catalog tooling: scripts/photo-data.mjs (the curated URL list), scripts/photo-data.catalog.mjs (allocator — deterministic shuffle, global pool, refill on exhaustion), scripts/dedup-photos.mjs (filename-keyed dedup, dry-run support), scripts/run-bulk-eval.mjs (BL-54a Claude Vision evaluation across all properties).
