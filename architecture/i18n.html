HouseZoom.ai — Internationalization (i18n) Architecture

Last verified: 2026-04-28 Status: Living Update triggers: src/i18n/, src/messages/, locale-related migrations Owner: Will (solo) — review on every commit touching update triggers

Last updated: 2026-04-28 Status: Reference document. Reflects state after BL-50 (currency/measurement/voice), BL-60 (UI string keying), BL-68 (API error keying), BL-120 (FR/DE/IT translations + org→user inheritance + browser auto-translate disabled). AR pending BL-121 (RTL).

1. Goals & non-goals

Goals

  • Single source of truth for translations (messages/en.json)
  • Org-level default locale, user-level override
  • Server-side rendering in the user's locale on first paint (no FOUC, no client-side flash)
  • Translated API error responses (server returns errorKey, client renders translation)
  • Country-agnostic architecture — adding a new locale is ~30min translation script run + smoke
  • Consistent locale handling across web (SSR Next.js) and native (Capacitor iOS)
  • Verified translations not overlaid by browser auto-translate machinery

Non-goals (current state)

  • AR / RTL support — separate ticket BL-121
  • Per-request DB read for locale resolution — deferred to BL-123 (cookie-derived suffices)
  • Dynamic content translation (property descriptions, listing text — agents author in their own language)
  • Email template translation (separate track, lower priority)
  • Locale-aware SEO / hreflang tags
  • AI-generated content respecting user locale (bug, filed as BG-72 — separate track)

4. Locale resolution chain

4.1 Web (SSR)

getRequestConfig in src/i18n/request.ts resolves locale per request:

1. NEXT_LOCALE cookie
   ↓ if absent
2. Accept-Language header (parsed for first match in supported locales)
   ↓ if no match
3. 'en' (default fallback)

The cookie is the primary signal; the chain only falls through when the cookie is missing. There is NO direct DB read in the resolver — this would require Supabase round-trips per request and cookie-based SSR auth that isn't wired (deferred to BL-123).

4.2 Cookie ↔ DB reconciliation

The cookie is kept in sync with the DB at exactly two boundaries:

  1. Login (/api/auth/sync-locale): server-side endpoint reads agent_profiles.language (fall-through to organizations.default_locale → 'en') and returns Set-Cookie: NEXT_LOCALE=&lt;resolved&gt;. Called immediately after signInWithPassword(), before router.replace('/dashboard'). The browser processes the Set-Cookie header before the destination request fires, so SSR sees the correct cookie on first paint.
  1. Logout: clearLocaleCookie() from src/i18n/locale-utils.ts clears NEXT_LOCALE cookie + localStorage before supabase.auth.signOut(). Called from all 4 signOut sites (useAuth.ts, pending-approval/page.tsx, dashboard/page.tsx, AdminNav.tsx).
  1. Defense-in-depth: LocaleDetector.tsx reconciles cookie ← DB on every authenticated mount. Catches edge cases where the cookie drifts from the DB (e.g. user changes language in another browser tab).

4.3 Native (Capacitor)

Native skips the cookie chain entirely. CapacitorIntlProvider.tsx reads language from localStorage on app launch, falls back to 'en', and loads messages from public/messages/&lt;locale&gt;.json.

Locale is set on first authenticated app launch via API call to read agent_profiles.language, then persisted to localStorage. Web cookie state is irrelevant on native.

6. Browser auto-translate suppression

Critical for any i18n'd web app. Browsers (Chrome, Edge, Safari with extensions) detect mismatches between page language and user's preferred language and offer auto-translation. This auto-translation rewrites the rendered DOM at runtime, after our verified translations have already loaded — overlaying our content with machine translation that's lower quality and untestable.

This was the root cause of the BL-120 IT-on-Chrome bug: Chrome detected &lt;html lang="it"&gt; ≠ Accept-Language: en-US, offered to auto-translate the page, and rewrote our verified Italian into machine English. FR and DE escaped because Will's Chrome had different per-language auto-translate preferences. Safari users never hit it because they don't have built-in auto-translate at the same DOM-overlay level.

Mitigation (in src/app/layout.tsx):

<html lang={locale} translate="no">
  <head>
    <meta name="google" content="notranslate" />
    {/* ... */}
  </head>
  <body>{children}</body>
</html>
  • translate="no" is the W3C-standard attribute for opting out of automatic translation
  • &lt;meta name="google" content="notranslate"&gt; is the Google-specific equivalent (Chrome's translate engine uses this)

Together they disable Chrome's auto-translate offer + Google Translate browser extension + Edge's translation feature + similar third-party tools.

This is the standard mitigation for any app that ships professionally-translated content. We never want machine translation overlaying our verified strings.

Side-effect: users who legitimately want auto-translation (e.g. a Spanish-speaking visitor on our French app) lose that capability. Acceptable tradeoff — the cost of allowing it (broken UI for everyone whose system locale doesn't match our supported locales) far exceeds the benefit (a few users get a worse machine translation).

8. Key naming conventions

8.1 Namespace organization

Top-level namespaces in en.json (and mirror locales):

shared.*        — cross-domain UI strings (buttons, labels reused everywhere)
shared.errors.* — auth/membership errors (BG-60 Phase 2)
errors.*        — API error keys (BG-68)
  errors.generic.*    — cross-domain shared errors
  errors.signing.*
  errors.deals.*
  errors.actions.*
  errors.leads.*
  errors.photos.*
  errors.admin.*
  errors.orchestrator.*
  errors.ai.*
  errors.market.*
  errors.<subdomain>.*  — for any new subdomain
enums.*         — enum code → label maps (BG-60 Phase 3)
nav.*           — navigation items (bottom nav, sidebar)
dashboard.*
properties.*
contacts.*
deals.*
signing.*
settings.*
... (one namespace per major feature/page)

8.2 Key path structure

&lt;feature&gt;.&lt;screen&gt;.&lt;element&gt; or &lt;feature&gt;.&lt;context&gt;.&lt;reason&gt;

Examples:

  • properties.list.heading → "Annonces" / "Angebote" / "Annunci"
  • errors.signing.fetchDocumentFailed → "Document non trouvé" / "Dokument nicht gefunden"
  • enums.dealStage.active → "Actif" / "Aktiv"

8.3 ICU placeholders

Always use ICU param syntax for variables. Translation script preserves them verbatim:

{
  "errors": {
    "generic": {
      "missingRequiredField": "Field {field} is required"
    }
  },
  "dashboard": {
    "welcome": "Bonjour, {name} !"
  },
  "properties": {
    "count": "{count, plural, =0 {Aucune annonce} one {# annonce} other {# annonces}}"
  }
}

ICU plural / select / number / date syntax all supported by next-intl.

8.4 Anti-patterns (avoid)

❌ Hardcoded English in JSX:

<button>Save Changes</button>          // wrong
<button>{t('shared.saveChanges')}</button>  // right

❌ String concatenation:

{t('greeting')} + ', ' + name           // wrong (FR word order breaks)
{t('greeting', { name })}                // right ("Bonjour, {name}")

❌ Translation key with embedded data:

t(`status.${activeStatus}`)              // fragile if statuses change
{t('status', { context: activeStatus })} // better — explicit ICU select

❌ Reading navigator.language to drive translation logic on client. Always use the active locale from the resolver chain. Browser language is only for unauthenticated first-visit hint, not authenticated rendering.

❌ Removing translate="no" on &lt;html&gt;. Browser auto-translate WILL overwrite verified translations. See §6.

9. API error response shape (BG-68)

All API routes return errors via apiError() / apiErrorFromThrow() from @/lib/api/errors:

// In an API route
import { apiError, apiErrorFromThrow } from '@/lib/api/errors';

export async function POST(req: Request) {
  try {
    const body = await req.json();
    if (!body.dealId) {
      return apiError('errors.signing.missingRequiredField', { field: 'dealId' }, 400);
    }
    // ... happy path
  } catch (err) {
    return apiErrorFromThrow(err);
  }
}

Response shape:

{
  "error": "Field dealId is required",         // English fallback (logs, legacy clients)
  "errorKey": "errors.signing.missingRequiredField",  // primary signal for client
  "errorParams": { "field": "dealId" },
  "details": { "..." }                          // optional, e.g. Zod fieldErrors
}

Client-side (translateError()):

const message = errorKey
  ? t(errorKey, errorParams)
  : (error || 'An unknown error occurred');

10. Translation pipeline

10.1 Adding a new locale

  1. Glossary — node scripts/i18n/generate-glossary.mjs --target &lt;locale&gt; produces messages/_glossary.&lt;locale&gt;.json. Will reviews ~35 terms; this is the most leveraged review gate.
  1. Translate — node --env-file=.env.local scripts/i18n/translate-locale.mjs --target &lt;locale&gt; produces messages/&lt;locale&gt;.json and messages/_review.&lt;locale&gt;.csv. Idempotent / resumable / delta-only — only fills missing or identical-to-EN keys. Cost: ~$1-2 per locale at Sonnet pricing.
  1. Add to CHECK constraints if not already present: ``sql ALTER TABLE organizations DROP CONSTRAINT organizations_default_locale_check; ALTER TABLE organizations ADD CONSTRAINT organizations_default_locale_check CHECK (default_locale IN ('en','fr','de','it','ar','&lt;new&gt;')); -- same for agent_profiles.language ``
  1. Smoke test — flip a test user's agent_profiles.language via SQL, log in, verify 5 surfaces in target language. Use Safari (system locale matches) AND Chrome (Accept-Language: en-US) to catch any auto-translate or hydration bugs.
  1. Capacitor rebuild — bash scripts/build-capacitor.sh to populate public/messages/&lt;locale&gt;.json.

10.2 Updating an existing locale (after adding new EN keys)

Same script handles this idempotently:

node --env-file=.env.local scripts/i18n/translate-locale.mjs --target fr
node --env-file=.env.local scripts/i18n/translate-locale.mjs --target de
node --env-file=.env.local scripts/i18n/translate-locale.mjs --target it

The script reads messages/&lt;target&gt;.json, computes the gap (missing keys + identical-to-EN-but-should-be-translated), and translates only the delta. Existing translations are never overwritten.

10.3 Translation prompt structure

Per-batch, ~50 keys at a time:

System:
  You are translating UI strings for a real estate CRM into <target_lang>.
  Domain: real estate brokerage, B2B. Tone: professional, terse, native-speaker.
  Country anchor: <FR=France+Romandie | DE=Germany+Deutschschweiz | IT=Italy+Ticino>.
  Glossary (use these exact translations):
  <glossary_entries>
  Rules:
  - Preserve ICU placeholders ({field}, {count}, etc.) verbatim
  - Preserve markdown/HTML if present
  - Output JSON only: {"key1": "translation1", ...}
  - Never translate proper nouns (HouseZoom, Bayut, etc.)
  - For DE: use formal "Sie", not "du"
  - For IT: use formal "Lei"

User:
  Translate these strings. Each key path provides context.
  <numbered_list_of_key_value_pairs>

ICU validation runs after translation — any response where {...} count differs from source is rejected and the key falls back to EN.

11. Database schema

11.1 organizations.default_locale

default_locale TEXT
  DEFAULT 'en'
  CHECK (default_locale IN ('en','fr','de','it','ar'))

Used as inheritance source for new agent profiles. Editable by org admins (UI deferred — currently SQL-only).

11.2 agent_profiles.language

language TEXT
  CHECK (language IN ('en','fr','de','it','ar'))

NULL means "inherit from org default". Set on user creation, editable via Settings → Preferences.

11.3 Deprecated path: agent_profiles.preferences-&gt;'language' (jsonb)

Pre-BL-120 path. Now deprecated; top-level language column is source of truth. BL-122 will migrate any remaining readers and drop the jsonb subfield.

12. Known issues and design tradeoffs

12.1 No per-request DB read for locale (BL-123)

Resolver chain stops at the cookie. If a user changes their locale in browser tab A, tab B doesn't pick it up until next reload. Acceptable given:

  • Locale changes are rare (settings page, not navigation flow)
  • Per-request DB read costs 30-100ms latency on every cached page render
  • Requires SSR cookie auth (@supabase/ssr) not yet wired

When SSR auth is wired (other reasons), revisit per-request DB read as the cleaner pattern.

12.2 AI-generated content doesn't respect user locale (BG-72)

Plan du jour, AI assistant chat, walkthrough narration — these call Claude API at runtime. The user's locale isn't passed in the prompt, so the model defaults to English. UI keys around the AI content render in target locale, but the AI content itself is English. Affects FR/DE/IT/AR equally.

Cross-cuts BL-38 (orchestrator), BL-85 (walkthrough), AI assistant chat, anywhere AI text reaches users.

12.3 Capacitor + native locale parity

Native iOS app uses a different locale path than web — localStorage instead of cookies, no DB integration in the resolver. Translations are bundled at build time (scripts/build-capacitor.sh:88 copies messages/*.json → public/messages/). Updates require a Capacitor rebuild.

This is acceptable today since native is single-user-per-device. Multi-user device + locale switching is hypothetical for native and not modeled.

12.4 Browser auto-translate disabled globally

§6 above. Side-effect: visitors whose system locale isn't in our supported set lose access to browser auto-translation. Acceptable tradeoff — auto-translate overlay broke verified IT translations on Chrome in BL-120, and the cost of allowing it (broken UI, debugging time-sinks) exceeds the benefit (a few visitors get worse machine translation).

14. Testing checklist for new locale work

When changing anything in the i18n stack, verify:

  • [ ] Server HTML correct: fetch('/dashboard').then(r =&gt; r.text()) contains &lt;html lang="&lt;locale&gt;" translate="no"&gt; and target-language strings
  • [ ] DOM matches: document.documentElement.lang === locale, visible text in target language
  • [ ] Cookie correct: document.cookie contains NEXT_LOCALE=&lt;locale&gt;
  • [ ] DB correct: SELECT language FROM agent_profiles WHERE user_id = ? matches
  • [ ] Login transition: log out, log back in → first paint is in correct locale (no flash)
  • [ ] Logout clears: log out → cookie removed, next user's locale is independent
  • [ ] User override: change in Settings → Preferences → persists across reload
  • [ ] Org inheritance: new user via invite → agent_profiles.language matches org.default_locale
  • [ ] User override beats org default: user changes locale → org default stays untouched
  • [ ] Fallback: temporarily delete a key → EN string appears
  • [ ] API errors: trigger 2-3 API errors (bad request, 404, validation) → errorKey translates correctly
  • [ ] No browser auto-translate offer: address bar shouldn't show a Translate icon when viewing the app in a non-system locale
  • [ ] Cross-browser: test on Safari AND Chrome
  • [ ] Cross-Accept-Language: test with system locale matching AND not matching the user's preferred locale

18. Absorbed tickets (BG-79b pass 1)

The i18n surface accumulated through several large campaigns. Each entry below captures the architectural significance — what invariant or pattern it established — not a changelog summary.

Audit-driven coverage campaigns

  • BG-50 — Agent Context Propagation: currency, measurement unit, locale (5 wave campaign across BG-50X foundation; 529 strings wired across 75+ files; coverage 80.2 → 94.1%, MCP-verified 0 regressions)

The largest i18n campaign of the project. Pattern: extend useAgentPreferences() hook to expose currency, measurementUnit, locale (and later phoneCountryCode per BG-50Z); plumb those values through every formatting site that touches money, area, or locale-specific phrasing. The "wire-vs-skip" decision is audit-driven: if the audit detector flags a string and it's user-visible content, wire it; if it's a brand name, emoji, dev-only string, enum code, Tailwind class, or HTML entity, the audit should classify-and-skip rather than flagging it as missing. BG-50 also surfaced the canonical lesson that duplicate top-level JSON namespaces silently override each other (last-key-wins) — a 40-key MISSINGINEN spike caught mid-batch and fixed by merging duplicate intel: and portalSimulator: blocks.

  • BG-46 — ViewingMode multi-return modal safety ([cross-bucketed: surfaced via i18n audit, but the underlying pattern is React render-tree discipline]) — files: src/components/shared/ViewingMode.tsx

When a component has multiple return statements per screen state (one per screen), a modal/panel render block must live inside EACH return tree, not just one. Discovered in BL-112 Pass 2 when QaPanel was rendered in only one of three screen returns — the panel would silently disappear on screen change. Two acceptable fixes: (a) hoist the panel render to the component root, (b) refactor multi-screen returns into a single return that switches on screen state. The audit-aware lesson: i18n coverage audits surface this kind of bug because they walk JSX trees per file, but the fix is React-pattern-level, not i18n-level.

  • BG-54 — admin.nav.autonomy MISSINGMESSAGE noise (closed in BG-50X Batch 2) — files: messages/{en,fr,de,it}.json, admin nav component_

A single missing key was firing 70+ MISSINGMESSAGE console errors per admin nav re-render. Pre-existing, not a regression — but a reminder that even one missing dynamic key creates console noise at render-frequency. Fix: add the key to all four locales. Static audit may still report unresolvabledynamic_t (a BG-59 pattern-a artifact); runtime resolution is what matters.

  • BG-57 — i18n audit script blind-spot fixes (4 detection patterns + classifyHardcoded value-collision bug) — files: scripts/audit/i18n-coverage-full.mjs

Audit script extended to detect: (1) JSX text adjacent to child spans (e.g. Buyer Agent &lt;span&gt;(optional)&lt;/span&gt; was missing the leading text), (2) top-level page &lt;h1&gt;/&lt;h2&gt; headings in non-component pages, (3) status-label ternaries (x ? 19.2 Fedlex adapter (BL-132)

src/features/compliance/adapters/fedlex-url.ts:

export interface FedlexUrlResult {
  url: string
  /** True if the URL points to an English Fedlex page (informational only per PublA art. 14). */
  informational_only: boolean
}

export function getSourceUrlForLocale(
  storedUrl: string,
  locale: string,
): FedlexUrlResult

Behaviour:

  • Empty / malformed / non-Fedlex URL → returned unchanged, informational_only=false.
  • Fedlex URL without a /de|/fr|/it|/en path segment → returned unchanged.
  • Unsupported locale (e.g. 'rm', 'ar') → swap to /de.
  • Supported locale → segment swapped; query / fragment / trailing slash preserved.
  • 'en' → segment swapped AND informational_only=true per PublA art. 14 (only DE/FR/IT are official languages of the Federal Publication; English Fedlex pages are advisory).

Stored URL convention: German canonical (e.g. https://www.fedlex.admin.ch/eli/cc/.../de). Federal rules ingested under BL-132 use this convention exclusively. Non-Fedlex URLs (e.g. https://www.svit.ch/...) pass through unchanged — non-mechanical sources don't get auto-translated.

Test coverage: 14 unit tests at tests/unit/fedlex-url.test.ts.

20.3 Sync mechanism

src/features/agencies/services/cantonLanguageSync.ts:

export async function syncOrgMembersLanguageOnCantonChange({
  orgId,
  oldCanton,
  newCanton,
  supabase,                  // service-role; bypasses RLS on agent_profiles
}: SyncOrgMembersLanguageInput): Promise<SyncOrgMembersLanguageResult>

Single-writer service. When organizations.canton changes:

  1. Compute oldDefault = getCantonDefaultLanguage(oldCanton) + newDefault = getCantonDefaultLanguage(newCanton).
  2. Same-default short-circuit: if oldDefault === newDefault (e.g. CH-ZH → CH-LU, both de), do nothing — not even a no-op UPDATE. Count org members for the skipped field, emit one audit log line, return.
  3. Real shift: UPDATE agent_profiles SET language = newDefault WHERE org_id = $orgId AND language = oldDefault. Members whose language differs from the old default are presumed to have explicitly overridden (via profile settings, login locale-cookie, or seed override) and are left untouched.
  4. Audit log via structured console.info: ``ts {event: 'canton_language_sync', orgId, oldCanton, newCanton, oldDefault, newDefault, updated, skipped, shifted, ts} `` Vercel-log-ingestible; future promotion to a real audit table is non-blocking.

The "explicit-override heuristic" is intentionally conservative — a member who happens to manually pick de while in a CH-BE org will be flipped if the canton moves CH-BE → CH-GE. Same outcome as if their language had been auto-set; documented edge case.
