BL-153 — Background Drift Audit Agents (Phases A/B/C)

Spec version: v0.1 (draft, awaiting Will review) Filed: 2026-05-22 Author: Claude.ai web (this session) Status: SPEC — not yet locked, not yet implemented

Provenance: This spec is the corrective action filed in response to two governance drift events surfaced 2026-05-22:

  1. D-340 drafted Phase 3 axis prompts using axis numbers (6/2/5) that did NOT match the canonical 10-axis DD checklist in dd-hardening-tracker.xlsx § Pillar 1 Detail § Section 3. The drift went undetected through D-341, D-342, D-343 — three downstream sessions — and would have continued indefinitely without manual audit.
  2. housezoom-dd-hardening-plan-v0.2.docx, referenced by the tracker as the source of the hardening plan, does not exist anywhere in the repo or on Will's Mac. Reference was treated as canonical for weeks.

Root cause of both: no automated cross-validation between paper-trail artifacts. Drift accumulates silently between sessions; only manual audit surfaces it, and only after damage.

Cost of drift: ~1h of Will's time + a chunk of confidence per event. At observed cadence (2 events in 2 weeks), ~50h/year.

Three nightly Claude-agent jobs running in GitHub Actions, layered:

All three phases run nightly at 04:00 UTC (after EU-late EOS commits land). Each writes a drift report to docs/audits/. Each opens a GitHub issue when 🔴 drift is detected. None auto-commits fixes — review remains human-gated.

Why three phases, not one

Phase A is the must-have. It's self-contained, ships in one PR, and would have caught the D-340 drift the night it happened.

Phase B is a 1-hour incremental extension of A — same workflow, additional checks. Defer if Phase A reveals A-class drift is enough surface area to monitor.

Phase C is structurally different — needs a read-only Supabase service role secret, needs its own dump+diff infra, has its own failure modes. Worth its own PR + ramp-up. Solves a different class of bug (the D-342 dual-column disaster) that Phases A/B cannot see.

The spec covers all three so that the infrastructure investment is amortized — same GH Actions account, same artifact-storage pattern, same issue-templating, same agent-prompt structure. Phase A pays for Phase B's hosting; Phase B pays for Phase C's scaffolding.

What the agent CANNOT check (out of scope for Phase A)

  • Anything live (Supabase, deployed app, MCP) — Phase C territory
  • Anything requiring browser drive — never; that's Will + Claude Desktop + Chrome MCP
  • Anything that requires interpretation of intent (e.g., "should this BG be 🔴 or 🟡?") — only mechanical consistency
  • Anything outside the repo (Vercel logs, Anthropic API console, etc.)

Workflow

.github/workflows/drift-audit.yml triggered by:

  • Nightly cron: 0 4 * * * (04:00 UTC)
  • On push to main touching docs/, ROADMAP.md, CHANGELOG.md, WAYS-OF-WORKING.md, package.json, docs/dd-hardening/dd-hardening-tracker.xlsx
  • On PR with same path filter (posts findings as PR comment instead of issue)
  • Manual: workflow_dispatch for ad-hoc runs

Agent runtime

Uses Anthropic's official claude-code-action (or equivalent) running claude-sonnet-4-6. Falls back to a retry on claude-opus-4-7 for any single run that returns a malformed JSON output.

Why Sonnet 4.6: drift audit is text reasoning over markdown files. Doesn't need Opus 4.7's deeper reasoning. Cost per run with Sonnet ≈ $0.50-$1.50; with Opus ≈ $2-$5.

Prompt

Lives at scripts/audit/drift-audit/PROMPT.md. Versioned (e.g., PROMPT-v0.1.md). Strict structure:

  • Role: "You are a paper-trail drift auditor for the HouseZoom repo. Your job is mechanical cross-validation of governance docs."
  • Allowed tools: read-only (view, grep, extract-text for xlsx). NO write tools. NO bash that mutates.
  • Output: ONE markdown file matching scripts/audit/drift-audit/REPORT-SCHEMA.md (template).
  • Hard constraints: no fabrication; if a check cannot run, emit a "check skipped" finding with reason. No recommendations beyond what's listed in the spec's check table.
  • Self-check: before emitting, re-read own output against the template. If schema invalid, retry once.

Schema

scripts/audit/drift-audit/REPORT-SCHEMA.md — frozen template the agent must match. Validated by a small CI step (scripts/audit/drift-audit/validate-report.mjs) that checks every finding has: severity, check ID, affected files, canonical value, observed value, recommendation. Run as part of the workflow; failure → workflow fails → re-run.

Cost model

Per nightly run:

  • Avg context: ~80KB markdown + ~30KB xlsx text dump = ~25k tokens input
  • Avg output: ~3-5k tokens
  • Sonnet 4.6 input ≈ $3/1M tokens, output ≈ $15/1M
  • Per run: $0.075 input + $0.075 output = ~$0.15
  • Nightly = ~$4.50/mo
  • Plus on-push runs: assume ~30/mo × $0.15 = ~$4.50
  • Plus retry overhead, opus fallback: ~30%
  • Phase A total ≈ $12/mo (earlier $30/mo estimate was conservative; revising down)

Phase B — Doc-reference validator (extends A)

Goal

Detect references to files that don't exist in the repo. Catches the missing-housezoom-dd-hardening-plan-v0.2.docx class.

Delta over Phase A

Same workflow, same agent, same schedule. Adds one new check class (#7 in Phase A's check table is the seed):

Check 7-expanded: for every markdown link, bare file reference, or path mention in any of the Phase A source files:

  1. Extract the path (regex against .md, .docx, .xlsx, .pdf, .sql, .mjs, .ts, .tsx, .cjs, .py, .json extensions)
  2. Resolve relative to repo root
  3. Check if it exists in git ls-files
  4. If not: emit drift finding 🔴 (phantom reference)

Two exception categories that don't count as drift:

  • Paths under node_modules/, .next/, dist/ (ignored)
  • Paths explicitly marked [deleted in D-NNN] or [deprecated] in surrounding text (the agent reads context)

Output

Same artifact as Phase A. Adds a new section: "Phantom file references" listing every (referencing file, referenced path, exists?) tuple where exists=false.

Phase C — Schema-vs-docs sync

Goal

Detect drift between live Supabase schema and DB-SCHEMA.md. Catches the D-342 dual-column class — schema acquires structures that documentation doesn't know about, or documentation describes structures that no longer exist.

Why this is structurally separate from A/B

  • Needs a read-only Supabase service role secret in GH Actions Secrets (security boundary)
  • Needs Postgres client in the runner (psql or pg)
  • Needs schema dump + diff, not just markdown reasoning
  • Failure modes are different (network failure, auth failure, schema-dump format change)

What it does

Nightly:

  1. Connect to Supabase via a read-only role (newly created — drift_audit_readonly) using pg_dump --schema-only --no-owner --no-privileges
  2. Normalize the dump (sort statements, strip timestamps, strip extension version numbers)
  3. Compare to a checked-in snapshot at docs/db-schema-snapshot.sql
  4. If diff detected: - Parse the diff into structured findings (column added/dropped/renamed, RLS policy changed, trigger added, etc.) - For each finding, check whether DB-SCHEMA.md reflects it - Emit drift report tagged 🔴 if structure changed but DB-SCHEMA.md didn't update; 🟡 if DB-SCHEMA.md mentions structures not in live schema; 🟢 if both align (no drift)
  5. Also re-runs the audit query from WAYS-OF-WORKING.md § SECURITY DEFINER Function Discipline and reports any NEEDS_HARDENING rows that aren't on the known-exempted list (PostGIS extension functions)
  6. Updates the snapshot file IFF Will reviews and approves via a follow-up PR (NOT auto-committed)

Sources of truth

SourceRole
Live Supabase via drift_audit_readonlyAuthoritative schema
docs/db-schema-snapshot.sqlLast-approved schema baseline
DB-SCHEMA.mdHuman-readable canonical reference

What it CANNOT do

  • Run any DDL
  • Run any DML
  • Read user data (read-only role gets USAGE + SELECT on information_schema + pg_catalog only; no row access)
  • Auto-commit a snapshot update — that's always a Will-reviewed PR
