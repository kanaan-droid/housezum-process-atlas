BL-153d — Atlas "Ways of Working" Tab (Governance Section)

Spec version: v0.1 (draft, awaiting Will review) Filed: 2026-05-22 Author: Claude.ai web (this session) Status: SPEC — not yet locked, not yet implemented Parent spec: docs/specs/BL-153-Spec-Background-Drift-Audit-Agents.md (BL-153 / Phases A/B/C)

Provenance: Referenced as out-of-scope in BL-153 spec. Filed as a separate spec because it's a generator-side change to scripts/generate-*-html.mjs + atlas index config, with no functional overlap with BL-153 A/B/C. Drafted same session to lock the design while context was fresh.

TL;DR

Add a "Ways of Working" section to the Process Atlas at:

kanaan-droid.github.io/housezum-process-atlas/wow/

This section renders governance docs (WOW, STANDING-OBLIGATIONS, ClaudeAccess, this spec, the BL-153 spec, future ADRs and governance docs) as HTML pages with cross-linking, plus a rolling view of the latest N drift reports from docs/audits/.

The Atlas already has 4 sections (/roadmap/, /architecture/, /data-environments/, /legal-compliance/) generated by 4 near-identical scripts. BL-153d is the trigger to do the long-deferred TD-26b refactor — collapse the 4 generators into 1 config-driven generator, then add the 5th section.

Total work: ~4-5h. Refactor cost gets amortized across the existing 4 sections + the new 5th + any 6th in future.

Why filing as a separate spec

Three reasons:

  1. Different scope. BL-153 is "validate the paper trail." BL-153d is "render governance docs to HTML." Same parent but orthogonal.
  2. Different urgency. BL-153 is corrective action for two recent 🔴 drift events. BL-153d is nice-to-have surface; doesn't block anything.
  3. Different actor. BL-153 implementation is Claude.ai web (prompts) + CC (workflow). BL-153d is purely CC + Will (HTML generator changes).

BL-153d depends on no other BL-153 phase, and vice versa. They can ship in any order.

Generator behavior

generate-atlas-section.mjs accepts a section ID:

node scripts/generate-atlas-section.mjs roadmap
node scripts/generate-atlas-section.mjs wow
node scripts/generate-atlas-section.mjs all       # All sections

Reads ATLAS_SECTIONS, looks up the config, runs the layout-appropriate rendering pipeline:

  • single-page layout: pre-existing roadmap pipeline (one big page with collapsible sections)
  • multi-page layout: pre-existing arch/data/legal pipeline (one HTML per MD + cross-link rewriting + landing page)
  • rolling layout (NEW): for time-series content like drift reports — landing page lists the last N entries with dates, links to each

augments extend a multi-page section with additional rolling-content subdirectories (the drift-reports use case).

Backward compat

package.json keeps the existing 4 npm scripts (docs:html:roadmap, etc.) pointing at the new generator:

{
  "scripts": {

Old scripts still callable, exact same I/O contract. No retraining needed.

The "Read these in order at session start" header anchors the WOW section's purpose. Session-start workflow becomes: "open Atlas → WOW tab → top 3 links."

Per-doc pages

Each markdown file in the sources array gets its own HTML page. Cross-links between them work (e.g., WAYS-OF-WORKING.md references STANDING-OBLIGATIONS.md → in HTML, the link rewrites to STANDING-OBLIGATIONS.html).

Drift report subdirectory (/wow/audits/)

Latest 7 drift reports rendered as HTML with a sub-index page listing them by date.

When the BL-153a agent runs nightly and writes a new report, the next regen picks it up automatically. Reports older than 7 stay in docs/audits/ but aren't surfaced in the Atlas (they'd flood the index). Filing them is the agent's job; surfacing is the generator's.

If Will wants more than 7, change limit in the config. 7 is the default for fresh-attention surface.

Atlas index integration

scripts/generate-atlas-index.mjs reads the same ATLAS_SECTIONS config. Currently lists 4 sections in the Atlas root index. After BL-153d, it lists 5.

No code change needed in generate-atlas-index.mjs — it's already config-driven. Just gets one more entry from the imported config.

CSS / visual language

Match existing Atlas Purple palette (#0f0f11 background, the established typography). Reuse .arch-diagram / .arch-diagram-caption classes if any of the governance docs include SVG diagrams (none currently, but BL-153 spec could grow them).

No new visual language. The WOW tab visually looks like the architecture tab.

Rollback

If the refactor breaks anything:

  1. Revert the package.json script changes (point back at old generators)
  2. Move generators back from _DEPRECATED/ to scripts/
  3. Delete generate-atlas-section.mjs + lib/atlas-section-config.mjs

Reversal is ~5 min via git.

The _DEPRECATED/ move (not delete) per Commandment #6 means rollback is always possible. This is intentional friction-prevention.
