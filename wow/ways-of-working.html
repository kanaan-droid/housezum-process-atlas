  1. Before adding or deleting ANYTHING in the DB schema, do a FULL DB schema scan first. This includes: adding a column, dropping a column, renaming a column, adding/dropping a constraint, adding/dropping an index, adding/altering an RLS policy, adding/altering a trigger or function, adding/dropping a table, changing a FK target. No exceptions. The cost of a 5-minute scan is far less than the cost of a redundant column / orphaned policy / silent shadow-write that lives in production for months and corrupts data. It will not happen again. The full scan is mandatory even if the change seems trivial, because the entire point is that the impact is not visible from the proposed change alone.

What "full schema scan" means (Commandment #11)

Before ANY schema change is proposed, executed, or merged, run ALL of the following and include the outputs in the migration spec / PR / session report:

  1. Column inventory across all tables. SELECT table_name, column_name, data_type FROM information_schema.columns WHERE table_schema = 'public' AND column_name ILIKE '%&lt;concept&gt;%' ORDER BY table_name; — replace &lt;concept&gt; with every keyword the change touches (e.g. for an ownership change: user_id, owner, assigned, agent).
  2. RLS policies referencing the column or table. SELECT schemaname, tablename, polname, polcmd, pg_get_expr(polqual, polrelid) AS using_expr, pg_get_expr(polwithcheck, polrelid) AS check_expr FROM pg_policy p JOIN pg_class c ON c.oid = p.polrelid JOIN pg_namespace n ON n.oid = c.relnamespace WHERE pg_get_expr(polqual, polrelid) ILIKE '%&lt;column&gt;%' OR pg_get_expr(polwithcheck, polrelid) ILIKE '%&lt;column&gt;%';
  3. Triggers and functions referencing the column. SELECT n.nspname, p.proname FROM pg_proc p JOIN pg_namespace n ON n.oid = p.pronamespace WHERE n.nspname = 'public' AND pg_get_functiondef(p.oid) ILIKE '%&lt;column&gt;%';
  4. Indexes involving the column. SELECT tablename, indexname, indexdef FROM pg_indexes WHERE schemaname = 'public' AND indexdef ILIKE '%&lt;column&gt;%';
  5. FK constraints involving the column. SELECT conrelid::regclass AS table, conname, pg_get_constraintdef(oid) FROM pg_constraint WHERE pg_get_constraintdef(oid) ILIKE '%&lt;column&gt;%';
  6. Repo-wide grep for the column name and any synonym. grep -rn "&lt;column&gt;" src/ scripts/ supabase/ --include="*.ts" --include="*.tsx" --include="*.sql" --include="*.mjs" --include="*.py" — both the column itself AND any obvious synonym (e.g. owner and assigned for ownership changes). Output goes in the migration spec, file:line referenced.
  7. DB-SCHEMA.md cross-check. Compare the live schema against docs/DB-SCHEMA.md. If they have drifted, file a 🟡 TD before proceeding with the new change.

The output of all 7 scans must be embedded in the migration spec OR the PR description OR the session report. If any of the 7 outputs reveals usage Will did not know about, STOP and re-spec — do not proceed on the assumption that "it's probably fine."

This commandment applies to CC, Claude.ai, Claude Desktop, and Will himself. No actor is exempt.

Seed Script Discipline (D-282 — distilled from BL-126)

Hard rules that apply to every seed/migration-style script before live execution.

1. DB-SCHEMA.md is a hint, not a contract

  • Live pg_constraint is canonical. DB-SCHEMA.md drifts and has been wrong on multiple occasions in 2026 (BL-126 hit 4 stale entries in one script's worth).
  • Run the canonical pg_get_constraintdef query in docs/data-environments/SEEDED-DATA.md § 5.1 before any insert-heavy work. The output is cached in § 5.2; refresh whenever a migration lands.
  • BG-83 (filed 2026-04-29) will add a CI/EOS audit script that diffs against live constraint state.

2. NOT NULL audits are table-scoped, not column-scoped

  • Lesson from BL-126: hit viewing_feedback.user_id NOT NULL, patched it, immediately hit viewing_feedback.org_id NOT NULL on the next iteration. Two iterations on the same table because the first audit only patched the column that fired.
  • Before inserting into a table the script touches for the first time, list every column from information_schema.columns WHERE table_name = '&lt;table&gt;' AND nullability. Populate every NOT NULL column. Optional columns can be NULL.

3. CHECK constraint failures cascade — audit the whole script in one pass

  • When a CHECK violation surfaces, also audit free-text-into-enum-column risk across the WHOLE script before re-running, not just the offending field.
  • BL-126 hit viewing_feedback.next_step enum CHECK after already iterating on leads.purpose, viewing_feedback.user_id, and viewing_feedback.org_id. A single full-script audit at iteration 2 would have prevented iterations 3-4.

4. Marker convention for seed data

  • Every seeded row gets [seed:&lt;scope&gt;-v&lt;n&gt;] appended to its notes field. Examples: [seed:eu-v1] (BL-126 EU locales), [seed:uae-v1] (BL-127 — pending UAE retrofit).
  • The marker is the idempotency anchor + rollback discriminator. Don't introduce a dedicated is_seed boolean column — that's schema noise.
  • Org-config UPDATEs (e.g. organizations.default_locale) are NOT marked — those are real correctness fixes that persist independent of the demo data lifecycle.

5. Required CLI flags

Every seed/migration-style script MUST support:

  • --dry-run — log projected inserts, write nothing. MANDATORY for first run (per the corpus-pollution lesson in userMemories).
  • --force — rollback (delete by marker) + re-seed in one command. The iterative-demo-prep ergonomic.
  • --rollback — clean only, no re-seed.
  • --locale=&lt;x&gt; (or scope analog) — narrow execution for debugging.

6. Defensive payload over minimal payload

When in doubt about whether a column is nullable, populate it. The cost of populating a column that turns out to be nullable is one line of seed data; the cost of NOT populating one that turns out to be NOT NULL is one full iteration cycle. BL-126's viewing_feedback insert went through 3 patch iterations; "populate every plausibly-NOT-NULL column" got us to clean on iteration 4.

7. Phase 0 CHECK constraint snapshot is mandatory (D-336 — distilled from BL-144 + audit)

Every seed script's Phase 0 reality check MUST satisfy three layers. Each layer alone is necessary but not sufficient.

Layer 1 — Consult DB-SCHEMA.md BEFORE writing the spec. Before claiming any column's value set in a spec, grep DB-SCHEMA.md for the table and read the documented CHECK clauses (most live at the column row's Notes column; master-constraint reference table starts ~L2364). If DB-SCHEMA documents a value set, use those values — do not invent. Anti-example from BL-144 (D-335): spec § 5.2 claimed service_provider for deal_participants.role; DB-SCHEMA L884 already documented the live CHECK as buyer/seller/listing_agent/buyer_agent/notary/bank/trustee/developer_rep/guarantor. Spec author didn't consult; 30-error live arc followed.

Layer 2 — Call scripts/lib/pg-constraint-snapshot.mjs to validate spec claims against live CHECKs. Two-mode pattern (PostgREST cannot reach pg_catalog, so the snapshot SQL runs in Supabase SQL Editor):

# Part 1 — print the SQL
node scripts/<seed-dir>/phase0-reality-check.mjs --print-constraint-sql > /tmp/sql.sql
# Will runs /tmp/sql.sql in Supabase SQL Editor, saves the JSON result to /tmp/snapshot.json
# Part 2 — validate spec claims against the snapshot
node scripts/<seed-dir>/phase0-reality-check.mjs --check-constraints=/tmp/snapshot.json

Hardcoded &lt;SEED&gt;_CLAIMS in the phase0 script enumerate (table, column, expectedValues[]) triples. validateSpecValues() exits non-zero on any unknown_values mismatch — fails fast BEFORE any live INSERT. Reference implementations: scripts/bl144/phase0-reality-check.mjs + scripts/bl138/phase0-reality-check.mjs. Tests: tests/unit/pg-constraint-snapshot.test.ts.

Layer 3 — Wrap every live INSERT with scripts/lib/check-error-detector.mjs. The 23514 runtime detector parses (table, constraint_name) from any Postgres CHECK violation and attaches structured fields to the error so the orchestrator can decide rollback strategy. Necessary even with layers 1+2 because layer 2 only validates value-set claims the spec author made — a column with a value-set CHECK the author didn't claim, or a multi-column composite CHECK (e.g. properties_rent_frequency_required_on_rent which BL-144 also hit), still surfaces at INSERT time. Tests: tests/unit/check-error-detector.test.ts.

Why all three layers exist. The CHECK-mismatch-at-live-insert class hit us three times (BL-131 v2 caught it pre-write, BL-138 wave 1 narrowly avoided due to table-scope luck, BL-144 30-error live arc). Root-cause audit at docs/dd-hardening/td-pg-constraint-snapshot-audit.md (D-335) documents both the tooling gap (PostgREST blocks pg_catalog) and the process gap (spec authors not consulting DB-SCHEMA). All three layers together close both.

Anti-pattern caught: a Phase 0 script whose header comment claims § 4.1 is covered but whose body never queries pg_constraint. BL-144's phase0 did exactly this — the header said "behavioural probes on columns" and that was actually § 4.2's description, not § 4.1's. The audit (D-335) calls this out explicitly. Header comments MUST match the body's actual checks (Commandment 9 — "every file gets a header comment explaining its purpose" implicitly demands accuracy).

Demo Data Realism (D-335 — distilled from BL-144 Gate 3)

When seed-script contacts represent real-world organizations (banks, notary offices, government bureaus, regulatory agencies, courts), apply the real-org + real-address + fictionalized-person pattern.

The rule

  1. Organization name = real. Use the actual name as it appears on the org's public website / register (e.g. Notariat, Grundbuch- und Konkursamt Riesbach-Zürich, UBS Switzerland AG, Zürcher Kantonalbank).
  2. Address = real. Web-verify the street, postal code, and city against the organization's official source (e.g. notariate.zh.ch, the bank's branch locator). Do NOT invent an address.
  3. Contact person = fictionalized. Never use the real office head / branch manager / department lead. The fictional name must be clearly distinct from the real one — not a transliteration or homophone.

Why all three components matter

  • Real org name anchors the data in plausible reality. A demo with Acme Notary Inc. reads as toy data; Notariat Horgen reads as production.
  • Real address withstands Phase 3 audit scrutiny. An auditor cross-checking the demo against the real notariate.zh.ch will find consistency, not divergence. A fictionalized address fails audit on first lookup.
  • Fictionalized contact person prevents (a) impersonation risk if the demo leaks, (b) liability under data-protection law (real persons in seed data require consent), (c) reputational damage to the real office head.

Anti-pattern (caught at BL-144 Gate 3)

The initial BL-144 spec proposed 3 notary contacts at fictionalized addresses:

  • "Dr. Martin Hürlimann, Notariat Horgen, Seestrasse 124, 8810 Horgen" — wrong (real address: Dorfplatz 1)
  • "Dr. Beatrice Naef, Notariat Riesbach-Zürich (Kreis VIII), Genferstrasse 11, 8002 Zürich" — wrong (real address: Kreuzstrasse 42, 8008; Genferstrasse 11 is in Kreis 2 Enge, wrong district)
  • "Dr. Andreas Brügger, Notariat Meilen, Bahnhofstrasse 27, 8706 Meilen" — wrong (real address: Dorfstrasse 81)

Plus a coincidental near-match: the fictionalized "Dr. Beatrice Naef" was too close to the real office head Beat Buxcel (shared "Beat-" prefix). Renamed to Dr. Cornelia Brunschwiler at Will's decision.

Procedure

When the seed script touches a contacts table:

  1. Identify real-org contacts. Per buildRow inventory, flag every contact whose contact_type='service_provider' (or equivalent) and whose role is "represents external organization" — typically notaries, bank reps, government officials, brokers.
  2. Web-verify each address before any INSERT. Surface the verification snippet (URL + 3-sentence excerpt) to Will at Gate 3 for sign-off.
  3. Fictionalize the contact person with a name that has zero phonetic/orthographic overlap with the real office head. If the script doesn't know who the real head is, search-engine the office for current personnel before locking the fictional name.
  4. Tag the contact in the seed data with a discoverable marker (e.g. [real-org real-address fictionalized-person] inline note + org_name field set to the real org).
  5. Document in the spec (§ contacts) the real-org list and the fictionalization rule, so reviewers see the discipline applied.

RLS Policy Discipline (D-283 — distilled from BG-84)

A misconfigured "service role bypass" policy on ai_action_cards and agent_daily_plans caused a P0 cross-tenant data leak. Hard rules below prevent recurrence.

1. Service-role bypass policies must always be TO service_role

Wrong (creates a global qual: true permissive policy that OR's with every other policy on the table):

CREATE POLICY "Service role full access" ON my_table
  FOR ALL USING (true);  -- ❌ defaults to roles = {public}

Right:

CREATE POLICY "Service role full access" ON my_table
  AS PERMISSIVE
  FOR ALL
  TO service_role        -- ✅ scoped to service-role JWT only
  USING (true)
  WITH CHECK (true);

2. Multiple PERMISSIVE policies on one table are OR'd together

If table T has policies A (auth.uid() = user_id) and B (true), then for a SELECT against T the row passes if A OR B — and B always wins.

When adding a permissive policy alongside an existing user-scoped one, the new policy is by default an expansion of access, not a restriction. Always verify the matrix of (policy, role) before shipping.

3. Audit query — must return zero rows for non-public tables

Before every release that touches RLS, run:

SELECT tablename, policyname, permissive, roles, cmd, qual
FROM pg_policies
WHERE schemaname = 'public'
  AND roles = '{public}'
  AND (qual = 'true' OR with_check = 'true')
ORDER BY tablename, policyname;

Expected matches are only:

  • Truly public read tables (market_*, building_stats, portal_* reference data) — explicit allowlist documented inline below.
  • Token-gated public-write paths (document_field_values, media_events, listing_outcomes, checklist_share_tokens) — the API route layer enforces the gate; the policy is the second line of defense.

Any other row in the result set is a potential cross-tenant leak. Investigate before merging.

pg client wire format ≠ PostgREST wire format (D-310)

Live-DB parity tests use the pg client (via tests/unit/__helpers__/db.ts). pg's defaults differ from PostgREST in two ways that matter for fingerprint parity:

  • Timestamps: pg parses timestamp (OID 1114) and timestamptz (OID 1184) into JS Date objects. PostgREST returns ISO 8601 strings. The fingerprint canonical contract assumes ISO strings (per § Compliance Acceptance Flow #1). Without intervention, s() on a Date produces Date.prototype.toString() ("Thu Apr 30 2026..."), which never matches the SQL canonical.
  • Even when configured to return strings, pg emits Postgres-native format (2026-04-30 00:00:00+00 — space separator, short TZ), not PostgREST/ISO 8601 (2026-04-30T00:00:00+00:00).

The fix lives in tests/unit/__helpers__/db.ts: register type parsers for OIDs 1114 + 1184 that convert pg's native format to ISO 8601 (pgTimestampToIso(val)):

  • replace(' ', 'T') for date/time separator
  • replace(/([+-]\d{2})$/, '$1:00') for short-TZ padding

JSONB is not affected — pg parses jsonb into JS objects, and JSON.stringify produces compact JSON matching the SQL jsonb_compact() helper byte-for-byte.

When adding a new live-DB parity test (e.g. another hash function in some future feature), use getTestPgClient() and the type parsers fire automatically.

Diagnostic discipline for parity test failures (D-310)

When a live-DB parity test fails, instrument before fixing. Two ways the standard console.log approach can mislead:

  • Jest in --ci mode buffers/suppresses console.log output. Use throw-based diagnostics instead — build a string payload and throw new Error(payload) when the divergence triggers. Jest always renders assertion error messages.
  • Don't remove the diagnostic in the same commit as the fix. Land the fix first, verify CI green, remove the diagnostic in the close-out commit. The first fix attempt for D-310 (commit 4c22082) bundled diagnostic removal with the fix; the fix was incomplete, and we lost a round-trip rebuilding the diagnostic (beacde4).

The diagnostic shape that worked for D-310: re-derive the JS canonical TWICE — once from the dbRow (what JS sees), once from a separate SELECT ... ::text query (what SQL canonicalizes from). If the two re-derivations differ, JS preprocessing diverges. If they're equal but the sha256s differ, the SQL function body diverges. This collapses three candidate causes (genuine drift / wire-format / fixture edge case) into a single boolean read on the CI log.

Test scoping discipline (BL-77)

  • Test IDs go on the row, not the parent container. data-testid="lead-row" on the &lt;tr&gt; itself, not on the &lt;tbody&gt;. Lets specs target one row deterministically when the list updates.
  • Sections need addressable anchors. data-testid="intel-cards-section" placed on the section that scopes Intel Cards on the dashboard, so specs can locator(...).getByText(...) within scope. Avoid global getByText when the same text could appear elsewhere on the page.

  • Cold-start specs go first or are isolated. A cold-start expectation in the middle of a serial run will fail because earlier tests warmed the page. Either run cold-start specs first, or mark them with their own project that bypasses storageState.

Migrations are idempotent by default (D-308 / TD-CI-SCHEMA-BASELINE-PROMOTE)

CI clean-slate replays every migration from tests/ci/schema-baseline.sql (a real pg_dump of staging) + the full supabase/migrations/ cascade. Every migration must be safe to re-run on a database that already represents its post-application state. Production runs each migration once via the Supabase Dashboard SQL Editor; CI runs them many times. CI is now the source of truth for "would this migration apply cleanly tomorrow."

The 6 patterns

Statement typeIdempotent form
CREATE TABLE xCREATE TABLE IF NOT EXISTS x
CREATE INDEX x / CREATE UNIQUE INDEX xCREATE INDEX IF NOT EXISTS x
CREATE FUNCTION xCREATE OR REPLACE FUNCTION x
CREATE TRIGGER x ON tblDROP TRIGGER IF EXISTS x ON tbl; then CREATE TRIGGER x ON tbl ...
CREATE POLICY "x" ON tblDROP POLICY IF EXISTS "x" ON tbl; then CREATE POLICY "x" ON tbl ...
ALTER TABLE x ADD CONSTRAINT name ...ALTER TABLE x DROP CONSTRAINT IF EXISTS name; then ALTER TABLE x ADD CONSTRAINT name ...
ALTER TABLE x ADD COLUMN yALTER TABLE x ADD COLUMN IF NOT EXISTS y

CREATE TYPE has no IF NOT EXISTS form — wrap in a pg_type check inside a DO $$ ... END $$; block.

When mechanical idempotency isn't enough — DO-block guards

Some statements reference state that has evolved away in production (column renamed, source table dropped, schema CHECK constraints added). Wrap such statements in a DO $$ BEGIN IF EXISTS (...) THEN &lt;statement&gt;; END IF; END $$; guard that probes information_schema for the precondition. Six concrete patterns shipped in D-308:

  • Source table no longer exists — INSERT INTO X SELECT ... FROM legacy_table where legacy_table was dropped. Probe: information_schema.tables. Example: 20260307_consolidate_viewings.sql.
  • Source table column renamed/dropped — old INSERT references compliance_rules.scope (renamed). Probe: information_schema.columns. Example: 20260315_checklist_compliance_rules.sql uses scope as a "still has old shape?" probe.
  • Index references dropped column — CREATE INDEX … ON documents(uploaded_at DESC) where uploaded_at was renamed to created_at. Same column-existence probe. Example: 001_initial_schema.sql:91-92.
  • Backfill INSERT without natural key — INSERTs from one table to another without ON CONFLICT support. Add WHERE NOT EXISTS (SELECT 1 FROM target WHERE &lt;key&gt;). Example: 20260310_deal_entity_v2.sql.
  • Seed INSERT without unique constraint — Wrap entire seed block in a "table populated for this scope?" check. Example: 20260308_portal_simulator.sql checks (portal=bayut, market=UAE) before seeding.
  • Pre-condition RAISE EXCEPTION — migration aborts if a CI-absent precondition is missing (e.g. kanaan@gmail.com user). Convert RAISE EXCEPTION to RAISE NOTICE; RETURN; so CI silently skips. Dev-loop guidance preserved in the NOTICE. Example: 007_seed_test_data.sql.

Every guarded statement carries an inline Idempotency sweep — TD-CI-SCHEMA-BASELINE-PROMOTE comment for greppability.

When to refresh the baseline

Refresh tests/ci/schema-baseline.sql from staging when a migration references a table that doesn't exist in the current baseline AND isn't created by an earlier tracked migration. That's the orphan-table case CI surfaces as relation "X" does not exist. The baseline is a snapshot of staging's public schema at refresh time; anything ALTERed thereafter in tracked migrations needs the baseline to know about its starting state.

How to refresh the baseline

bash scripts/refresh-ci-baseline.sh

Prompts for the staging Postgres password, runs pg_dump against the Session Pooler (IPv4 — see § "GitHub Actions → Supabase = Session Pooler"), writes tests/ci/schema-baseline.sql.new. Then apply 4 PG15 compat fixes:

# 1. strip pg_dump 18.x metacommands
sed -i '' -E '/^\\(restrict|unrestrict)/d' tests/ci/schema-baseline.sql.new

# 2. strip PG17+ GUC
sed -i '' -E '/^SET transaction_timeout/d' tests/ci/schema-baseline.sql.new

# 3. defuse public schema collision
sed -i '' -E 's/^CREATE SCHEMA public;$/CREATE SCHEMA IF NOT EXISTS public;/' tests/ci/schema-baseline.sql.new

# 4. requalify Supabase-namespaced extension functions
sed -i '' -E 's/extensions\.(gen_random_bytes|uuid_generate_v4)/public.\1/g' tests/ci/schema-baseline.sql.new

Then promote, preserving the prior version as a rollback:

mv tests/ci/schema-baseline.sql tests/ci/schema-baseline.sql.old-stopgap-$(date +%Y%m%d)
mv tests/ci/schema-baseline.sql.new tests/ci/schema-baseline.sql

Push to a feature branch, watch CI. Iterate on whatever drift surfaces — the iteration pattern is "fix the failing migration" with a sub-rule "bundle confirmed-failing siblings (same root cause + concrete grep evidence) when the prompt's per-migration discipline allows".

Why this exists

Pre-D-308 the migration cascade ran successfully in production but failed CI's clean-slate replay because production schema had drifted away from migration history (orphan tables, columns renamed, constraints added inline). The fix wasn't to abandon CI; it was to (a) make CI start from a real production-schema snapshot, and (b) make every migration idempotent against post-application state. Both halves are required — only the second is on every future migration's author.

Supabase email case-normalization mismatch

auth.users.email is automatically lowercased by Supabase per RFC 5321. But application-layer inserts to agent_profiles.email preserve the original case provided by the client. This means auth.users WHERE email = 'will+AG-FR-jean@...' returns 0 rows (stored as will+ag-fr-jean@...) but agent_profiles WHERE email = 'will+AG-FR-jean@...' returns 1.

Defenses installed 2026-04-22:

  1. New BEFORE INSERT OR UPDATE trigger on agent_profiles that auto-lowercases email: ``sql CREATE OR REPLACE FUNCTION normalize_agent_profile_email() ... CREATE TRIGGER normalize_agent_profile_email_trigger ... ``
  2. All seed scripts must .toLowerCase() email before lookup against auth.users. See seed-fr-properties.mjs line 29 for reference.
  3. Always use ILIKE or LOWER() when searching email columns from SQL. PostgreSQL LIKE is case-sensitive; an uppercase pattern against lowercased Supabase Auth data returns 0 rows even when records exist. D-315 burned ~15 minutes on this during BL-138 MCP verification. Diagnostic queries against auth.users.email, auth.identities.identity_data-&gt;&gt;'email', or agent_profiles.email should never use LIKE — always ILIKE.

Atlas generator discipline (D-323 — lesson from D-321→D-322→D-323 arc)

Every user-sourced field injected into HTML via runtime template literals (in the embedded &lt;script&gt; block of scripts/generate-roadmap-html.mjs) MUST be wrapped in escapeHtml(). Unescaped &lt;, &gt;, &amp;, or quote chars in ROADMAP.md content silently break the DOM tree — corruption appears as misplaced/missing columns or stray text, NOT as a parser error.

Build-time JSON.stringify(DATA).replace(/&lt;/g, '\\u003c') (lines 168-169) safely encodes data INTO the script tag, but the browser then de-escapes back to raw &lt; at parse time. The unsafe step is the subsequent ${item.desc} interpolation into innerHTML. escapeHtml() runs at that interpolation site to keep the DOM tree intact.

Verify after any generator change by opening the regenerated HTML in a browser and running in DevTools:

[...document.querySelector('.kanban').children].filter(el => el.classList.contains('column')).length

Debt Ceiling

The codebase and architecture docs must remain in sync. To prevent silent drift:

  2. Header freshness. For each file in docs/architecture/, check the "Last verified" date. If older than 30 days AND the trigger files have been touched since that date, fail with a list of stale docs and the commits that should have updated them.

Why this works (post-mortem of the old protocol)

The previous "Living Documentation Protocol" said "update ARCHITECTURE.md every session." It died because:

  • Vague trigger. "Every session" — which session? what changes count? CC silently degraded enforcement until it stopped.
  • Unbounded scope. ARCHITECTURE.md only grew. Every update required re-reading the whole file. Cost compounded.
  • Trigger at the wrong moment. "Session end" is when everyone is tired. By the time you remember, you've forgotten what changed.

Country-Agnostic Architecture

HouseZoom operates across UAE, France, Switzerland, and Italy. All country-specific UI must use lookup map patterns, never if/else chains.

Principle

Country-specific values (labels, prefixes, currencies, regulatory terms) live in a single lookup map constant. Components read from the map using the agent's operating_country. If a key is missing, fall back to a sensible default.

Patterns

LICENSE_META — country-specific license labels and prefixes:

const LICENSE_META: Record<string, { label: string; prefix: string }> = {
  UAE: { label: 'RERA License', prefix: 'RERA-' },
  France: { label: 'CPI License', prefix: 'CPI-' },
  Switzerland: { label: 'BF License', prefix: 'BF-' },
  Italy: { label: 'Patentino', prefix: 'Pat.-' },
}
// Usage: LICENSE_META[operatingCountry] ?? LICENSE_META['UAE']

setAgentCurrency — module-level singleton for currency resolution outside React context:

// In lib/utils.ts
let _agentCurrency: string | null = null
export function setAgentCurrency(c: string) { _agentCurrency = c }
// formatCurrency resolution: explicit arg > _agentCurrency > fallback

Phone country codes — dropdown options tied to operating country default.

Rules

  • Never hardcode "AED", "RERA", or any single-country value in components
  • Every country-specific string must have a lookup map or i18n key
  • Default to UAE values only as a fallback, not as the primary path
  • When adding a new market, add one entry to each lookup map — no component changes needed

The "No Hardcoding" Principle (Non-Negotiable)

Never hardcode market-specific, environment-specific, or configuration values in components.

This includes:

  • Currency codes ('AED', 'CHF', 'USD') → use orgCurrency from useOrg()
  • Country names or codes → use lookup maps
  • License labels ('RERA', 'CPI') → use LICENSE_META map
  • API URLs → use environment variables
  • Pixel values that depend on device/platform → use CSS variables or env()
  • Any value that differs by market, org, or device

The test: If you change markets tomorrow (UAE → France), would this value need to change? If yes → it must come from config, not be hardcoded.

Violations discovered late cost 10x more to fix than violations caught at write time. CC must flag any hardcoded market-specific value before committing.

Currency Rule (Non-Negotiable)

For repeatable, well-scoped batches (e.g. "wire surface X using audit-output/Y.csv"), prefer the no-gates model over the Step 0 gate model:

Service-layer i18n (BG-60)

Pure service files (src/**/services/*.ts, src/lib/*.ts — anything not a hook or component) cannot call useTranslations(). They communicate translatable content to the UI layer via two conventions:

For error throws (Shape A):

Services throw i18n: prefixed keys via the helper:

import { throwI18nError } from '@/lib/i18n/throwI18nError'

throwI18nError('deals.errors.failedToLoad')
throwI18nError('deals.errors.deleteFailed', { id: dealId })

Catch sites translate via the partner helper:

import { translateError } from '@/lib/i18n/translateError'

const t = useTranslations()
try {
  await fetchDeals(orgId)
} catch (err) {
  setError(translateError(err, t))
}

NEVER do setError(err.message) for errors originating from our service files — it surfaces the raw i18n:... key to users. The audit script (BG-59 follow-up) flags this pattern via Detector A.

NEVER do throw new Error('English literal') in service files — always use throwI18nError. The audit script flags this pattern via Detector B.

For const-map labels (Shape B):

Services export raw enums (ROOM_TYPES, INTENT_CARD_TYPES, etc.) as as const arrays or unions. Display labels live in messages/en.json under appropriate namespaces (e.g. rooms.labels.*, intentCards.labels.*). Translation happens at render time in the consuming component, NEVER inside the service.

// ❌ DO NOT — service exports labels
export const ROOM_TYPE_LABELS = { living_room: 'Living Room', ... }

// ✅ DO — service exports keys, en.json holds labels
export const ROOM_TYPES = ['living_room', 'master_bedroom', ...] as const
export type RoomType = typeof ROOM_TYPES[number]

// Component:
const t = useTranslations('rooms.labels')
<span>{t(roomType)}</span>

Why: services run in client, server, and cron/webhook contexts — useTranslations only works in the first. Stable i18n keys also give us clean Sentry/Datadog grouping (errors aggregate by key, not by fluctuating English string).

See docs/specs/BG-60-Spec-Service-Layer-i18n.md for the full strategy.

Every LLM call must read its model from a per-call env override that falls back to the default constant defined in the service's types/config file.

Fingerprint contracts (BL-136 hotfix #4 — 2026-05-01)

When an integrity hash is computed in two places (JS app side + SQL view side), one of them must be the spec, byte-for-byte. Drift between the two is a production correctness bug, not a stylistic difference.

1. JS canonicalization is the contract

The JS implementation at src/features/admin/services/ruleFingerprint.ts defines the canonical format. SQL functions mirror it byte-for-byte. PostgREST's natural serialization is the reference:

  • Timestamps: ISO 8601 with explicit timezone offset (2026-04-30T00:00:00+00:00) — what PostgREST returns to JS.
  • JSONB: no whitespace between structural tokens ([{"k":"v"}], not [{"k": "v"}]) — what JSON.stringify produces.
  • Object key order: alphabetical (jsonb's storage order) — what both PostgREST and jsonb_each traversal yield.

compute_rule_fingerprint(compliance_rules) in the BL-136 hotfix #4 migration uses to_jsonb(timestamptz) for ISO format and a recursive jsonb_compact() plpgsql helper for whitespace-free serialization. If ruleFingerprint.ts ever changes, ship a new SQL migration that updates the function body in lockstep.

2. Hardcoded fixture test is mandatory

A unit test with a frozen canonical string + frozen sha256 hex must run unconditionally in CI. If either the JS function or the SQL function changes, the fixture diverges and the test fails until both sides are realigned. Live-DB parity is additional, never a substitute. See tests/unit/ruleFingerprint.test.ts § "hardcoded regression fixture (BL-136 hotfix #4)" for the pattern.

3. Skipped tests are debt that bites in production

TD-COMPLIANCE-FINGERPRINT-CI-PARITY was filed at BL-136 Phase 1 with a describe.skip on the live-DB parity test (no local Supabase in CI). Hotfix #4 was the bill — every accept landed as stale_accepted in production because the JS hash and the SQL hash diverged on timestamp + JSONB whitespace. Wire local Supabase or a live RPC parity check to CI before any future hash-using feature lands.

Migration idempotency contract (BL-132 D-312 hotfix lesson)

Every schema migration that touches existing data must satisfy a strict idempotency contract so the file can be re-run on any environment (production, CI replay, fresh dev DB, staging snapshot restore) without error. Three concrete patterns now mandatory:

  1. Phase 0 verification logs state, does not RAISE EXCEPTION on column already exists. Use RAISE NOTICE for soft expectations (column may or may not exist — first run vs re-run).   1. All ALTER TABLE ADD COLUMN/ADD CONSTRAINT guarded. Use ADD COLUMN IF NOT EXISTS for columns; wrap ADD CONSTRAINT in DO $$ ... AND (canton IS NULL OR canton &lt;&gt; 'CH-ZH') rather than unconditional UPDATE. Same applies to data-cleanup UPDATEs that fix legacy values: UPDATE ... // In the browser console of the target tab:
const ANON = '<NEXT_PUBLIC_SUPABASE_ANON_KEY>';
const r = await fetch('https://<project>.supabase.co/auth/v1/token?grant_type=password', {
  method: 'POST',

const authPayload = {
  access_token: data.access_token,
  refresh_token: data.refresh_token,
  expires_at: Math.floor(Date.now()/1000) + data.expires_in,
  expires_in: data.expires_in,

// Then navigate to /admin/<whatever> — app picks up the session.

SECURITY DEFINER Function Discipline (D-315)

Every SECURITY DEFINER function in the schema MUST:

  1. Set explicit searchpath: SET search_path = public, pg_temp or whatever subset the function actually needs. Without this, the function executes with the caller's searchpath — which for service callers (GoTrue, pg_cron, postgrest) often does not include public, causing unqualified table references to fail at name resolution.
  2. Schema-qualify all table references: UPDATE public.agent_profiles ... not UPDATE agent_profiles .... Belt-and-suspenders with #1.
  3. Document the security boundary: function header comment explaining why SECURITY DEFINER is needed and what privilege escalation it enables.
  4. PostGIS extension functions are out of scope. public.st_estimatedextent (3 overloads) and any other functions installed by CREATE EXTENSION postgis will continue to flag as NEEDS_HARDENING in the audit query forever — proconfig is NULL on extension-installed functions and we cannot reissue CREATE OR REPLACE FUNCTION on them without forking them from the extension (next ALTER EXTENSION postgis UPDATE would then either fail or silently revert our changes). When the audit re-runs, expect these 3 rows to remain NEEDS_HARDENING; do NOT include them in hardening migrations. This rule generalizes: any SECURITY DEFINER function in public that is owned by an extension (check pg_depend for deptype='e') is excluded for the same reason.
