Standing Obligations

Cross-session recurring obligations. Read at session start. Update "Last touched" at EOS. File location: docs/sessions/STANDING-OBLIGATIONS.md (general-purpose, permanent home for cross-session obligations).

Provenance: D-338 (filed 2026-05-19 in Pillar 1 Phase 3 warm-up after second consecutive miss on dd-hardening-tracker.xlsx updates).

How to use

At session start: read the Open Obligations table. For each row, check whether its trigger applies to the work planned this session.

At EOS: for each open obligation, EITHER:

  • Update its "Last touched" column to today's date (if it was actioned), OR
  • Add a one-line note to the SOS explaining why it wasn't actioned (e.g. "no Pillar 1-5 work this session")

Soft enforcement: scripts/docs-check.mjs Check 4 parses this file and surfaces a soft warn if a commit appears to match a trigger but the corresponding row's "Last touched" wasn't bumped in the same commit. Check 4 NEVER fails — it only reminds.

Standing Obligation #8 — Functional Guide maintenance (detail)

Filed 2026-05-27 (BL-160 / D-364, Atlas Functional Guide section). Any session that ships user-visible behavior change updates the relevant docs/functional-guide/&lt;area&gt;.md:

  • New feature on an existing area → update §3 (How it works), §5 (Markets variance if applicable), §7 (Related tickets row), and the change log.
  • New area entirely → create a new &lt;area&gt;.md following the compliance.md template; add a config entry to scripts/lib/atlas-section-config.mjs; regen Atlas.
  • Edge case discovered in the wild → add a row to §6b Edge Case Catalog.
  • Behavior change resolves an open question → update that row's status to "Confirmed".
  • Standard regen → after any MD change: node scripts/generate-atlas-section.mjs functional-guide (or npm run docs:html:functional-guide), then commit + push from ~/Apps/housezum-process-atlas/.

Triggers a paper-trail check at EOS like any other standing obligation. Skipped only when session work is purely internal (refactors, infra, doc-only) with no user-visible surface change.

SOS template requirement (added 2026-05-27)

Every SOS file authored at EOS MUST include a "Functional Guide updates this session" section. The section is a small table with exactly these columns:

| Area touched | Guide file | Update made | Reason |

Allowed entries:

  • One row per functional area where this session updated the corresponding &lt;area&gt;.md. Cite the guide file path and the change made (e.g. "§6b row added (block-state advance behavior)").
  • A single row marked "(none)" / "—" / "—" / "Session was [infra-only / paper-trail-only / no user-visible change]" if no guide update applies.

The section is never omitted. An SOS with no "Functional Guide updates this session" section is incomplete. If you genuinely have nothing to update, you fill the row with the "none" version above — that's the discipline: positive confirmation either way.

Why this exists

The Functional Guide initiative (introduced session 8, 2026-05-27) is high-ROI but only if it stays current with actual behavior. Without an EOS trigger, guides drift. Without an SOS template requirement, the EOS trigger relies on memory. This amendment moves Obligation #8 from policy to enforced ritual.

If the SOS template requirement is bypassed for 2+ consecutive sessions without legitimate cause (e.g. all sessions genuinely infra-only), escalate to Layer 3: extend scripts/docs-check.mjs to fail-soft when a session SOS file is committed without the "Functional Guide updates this session" section. Until then, lightweight discipline.
